## Finance & Accounting — feature-level gap analysis (9 of 16 submodules)

Every row below is checked against the actual `erp_*` tables shipped in Wave 49
(`src/lib/db/schema.ts`), not assumed.

### General Ledger (`02`, GL001-GL015)
| Feature | VERIDIAN status |
|---|---|
| Company ledger, COA, journal entry create/edit/post | **Built** (`erpAccounts`, `erpJournalEntries`, `erpJournalEntryLines`) |
| Reverse journal | Schema supports it (draft/submitted/cancelled), no reversal *service* written yet |
| FX journal | `erpCurrencies` + `erpExchangeRates` exist as primitives; no currency/rate field on `erpJournalEntryLines` itself yet |
| Open/Close Period | **Gap** — `erpFiscalYears.isClosed` exists but there is no sub-period (monthly) table at all, no module-level lock |
| Trial Balance, P&L, Balance Sheet, GL Report | **Gap** — pure service-layer work sitting on schema that already exists (sum `erpJournalEntryLines` by account/rootType). Highest-value next wave: without this, the scaffold cannot produce a single financial statement. |
| Audit Trail | VERIDIAN's `audit_logs` system (Wave 7, immutable, org-scoped) is mature but not yet wired to ERP transactions specifically |
| AI Account Suggestion | **Gap**, but VERIDIAN's Prompt OS/LLM infra makes this a thin wrapper, not new infrastructure, once wanted |

### Chart of Accounts (`03`, COA001-COA010)
Root/child accounts, edit, categories: **built** (`erpAccounts` self-FK tree, free-text `accountType`). Gaps: account merge, opening-balance-as-its-own-flow (only a `isOpeningEntry` boolean on journal entries, no dedicated wizard), statutory account mapping, multi-company COA (VERIDIAN's org model is single-entity-per-org; no sub-"company" concept for group consolidation), semantic account search.

### Fiscal Year & Periods (`04`, FY001-FY010)
Fiscal year create/close: **built** (`erpFiscalYears`). **Real, meaningful gap**: no accounting-period (monthly) sub-entity at all — no open/close/reopen, no per-module period lock, no year-end wizard. This directly blocks the Trial Balance/P&L gap above from being date-range-safe in production (nothing stops posting into a "closed" year today because there's no period grain to lock).

### Journal Entries (`05`, JE001-JE010)
Core CRUD/approval/post: **built**. Gaps: recurring journals (no recurrence schema), journal templates (no template table), bulk CSV import (VERIDIAN has a general ingestion pipeline elsewhere, not wired here), journal attachments (VERIDIAN has generic document attachment infra elsewhere, not linked to `erpJournalEntries`).

### Accounts Payable (`06`, AP001-AP012)
Vendor master, PO, purchase invoice: **built** (`erpSuppliers`, `erpPurchaseOrders`, `erpPurchaseReceipts`, `erpPurchaseInvoices` — the actual primitives for 3-way matching already exist as separate linked tables). Gaps: no 3-way-match *service* logic yet, no vendor credit note table, no payment-run/proposal batching (only a generic `erpPaymentEntries` row), no AP aging/ledger report, no India-specific TDS logic beyond the generic `erpTaxTemplates` (worth flagging given VERIDIAN's India-compliance-first customer base — CA firms will expect TDS, not just GST).

### Accounts Receivable (`07`, AR001-AR015)
Customer master, sales invoice: **built** (`erpCustomers`). **Confirmed gap** (checked the actual columns): no credit-limit field on `erpCustomers` at all. Other gaps: no sales credit note table, no collection/reminder workflow (this is exactly the CSV's own named "AI collection scoring" opportunity — a natural fit for VERIDIAN's existing Worker Agent/Prompt OS infra rather than new infrastructure), no AR aging/ledger report.

### Banking (`08`, BK001-BK015)
Bank account master: **built** (`erpBankAccounts`, confirmed it already has `currencyId` — multi-currency accounts are *not* a gap, I was wrong to assume otherwise before checking). **Real gaps**: no bank statement import at all (the CSV explicitly names ISO20022/MT940/CAMT.053/UPI/NEFT/RTGS/IMPS — see GitHub candidates below), no reconciliation logic/table, no cheque management/printing, no account-freeze status.

### Cash Management (`09`, CM001-CM015)
**Entirely unbuilt** — zero cash-specific tables in Wave 49 (petty cash, cash receipt/payment, cash count, cash forecast, treasury position, daily cash closing all missing).

### Cost Centers (`10`, CC001-CC015)
**Mostly unbuilt, but not from zero**: `erpJournalEntryLines.costCenter` already exists as a free-text tag on every GL line (a lightweight hook, not a real dimension). Missing: an actual `erp_cost_centers` master table with hierarchy, budget-by-cost-center, allocation rules, variance analysis. VERIDIAN already has `departments` and `projects` tables from prior GRC/PMS waves that are natural mapping targets for cost-center dimensions rather than needing invention from scratch.

## What's needed for a genuine mid-size ERP (Finance), ranked

1. **Accounting periods (open/close/lock)** — currently zero schema. Blocks everything below from being safe.
2. **Financial reports (Trial Balance, P&L, Balance Sheet, AR/AP aging)** — schema is ready; this is pure service-layer aggregation and the single highest-value next wave.
3. **Cash Management** — entire submodule, zero schema.
4. **Cost Center dimension table** — upgrade the existing free-text tag into a real master table + link to `departments`/`projects`.
5. **Bank statement import + reconciliation** — see GitHub candidates below; don't hand-roll parsers for MT940/CAMT.053/ISO20022.
6. **Sales/Purchase credit notes** — currently zero schema on either side.
7. **Service-layer logic on top of existing schema**: 3-way matching, recurring journals, journal templates, reversal flow, payment-run batching.
8. **India-specific TDS** on top of the existing generic tax-template schema (GST already has a natural home there; TDS does not yet).

## GitHub candidates for reuse (step 5/6 — don't build, integrate)

Targeted at the concrete gaps above, license-checked, not "search for an ERP":

- **Bank statement formats (MT940/CAMT.053/ISO20022)**: `mt940` (npm, MIT) and `camt053-parser`-style packages parse these formats directly into structured transaction lists — this is exactly the "import bank statement" gap (BK006) and should be pulled in as a dependency, not hand-rolled regex/XML parsing.
- **India payment rails (UPI/NEFT/RTGS/IMPS) reconciliation**: these arrive as bank-specific CSV/Excel exports in practice more often than a single standard format; no single dominant open-source parser — likely needs a thin per-bank-format adapter layer over whichever CSV a customer's bank actually exports, not a generic library.
- **OCR invoice/receipt parsing** (named as an AI opportunity in AP CSV, JE010): Tesseract.js (Apache-2.0) is already a common Node-ecosystem choice; VERIDIAN's own Prompt OS + vision-capable OpenRouter models are actually a stronger fit here than a dedicated OCR library, since the extraction is semi-structured, not fixed-form.
- **Double-entry bookkeeping validation reference**: not adopting code, but `medici` (npm, a well-known double-entry ledger library) is worth reading for its balance-validation approach as a design reference the way ERPNext's doctypes were used for Wave 49 — same "research, don't copy" discipline.

### Summary: Real Gaps and Build-vs-Borrow Recommendations

Grounding: ERPNext doctypes and field lists were pulled directly from `frappe/erpnext` (`develop` branch) — e.g., `Lead`/`Opportunity` field sets, `Sales Order` commission fields (`commission_rate`, `total_commission`, `amount_eligible_for_commission` — flat fields, no payout engine), `Pricing Rule`/`Promotional Scheme` doctypes, and confirmation that **no dedicated RMA doctype exists** (returns use an `is_return` flag, not a workflow). Odoo's `addons/crm` and `addons/sale` model lists confirm `crm_lead_scoring_frequency` (rule-based scoring exists) and that pricing lives in `product_pricelist_item`.

**Top gaps, ranked by priority:**

1. **No pricing/discount rule engine (SL004)** — both Quotation and Sales Order items appear to be flat-rate; this blocks any tiered pricing, promotions, or margin control before invoicing goes live. Critical path for quote-to-cash.
2. **No RMA/returns workflow (SL006)** — a real gap even ERPNext under-serves (flag-based, not a workflow); building this well now is a differentiation opportunity rather than catch-up.
3. **No lead scoring beyond manual stage (CRM002)** — pipeline exists but has zero prioritization signal, a natural fit for VERIDIAN's AI-native positioning.
4. **No commission tracking (SL007)** — ERPNext's own implementation is thin (three fields, no ledger), so this is a genuine open problem across the industry, not just for VERIDIAN.
5. **No CPQ, campaign management, or customer portal** — lower urgency for CA-firm/consultant tenants (low product-configuration complexity) but blocking for any product-reselling tenant.

**GitHub candidates for narrow sub-problems (not full-suite adoption):**
- **Lead scoring**: no strong narrow npm library exists — most "lead scoring" packages are commercial SaaS wrappers. Recommend building a simple weighted-rules + logistic-regression scorer in-house using activity/engagement features; reference Odoo's `crm_lead_scoring_frequency` *design* (recency/frequency counting), not its code.
- **Pricing rules engine**: `json-rules-engine` (MIT, npm, ~2.5k stars) is a solid generic rules engine usable to implement tiered/conditional pricing without building a DSL from scratch.
- **RMA/returns workflow**: no good narrow open-source library exists — return-state-machine logic is domain-specific enough (partial returns, restocking, credit notes) that this needs to be designed and built in-house, informed by reading ERPNext's `is_return` handling and SAP's documented returns process as references only.

## Summary

Both modules are schema-only scaffolds — real tables exist for the "record-keeping spine" (suppliers, POs, receipts, invoices, warehouses, items, ledger, reconciliation) but almost none of the workflow or calculation logic that makes an ERP usable in production.

**Top gaps, ranked:**

1. **No inventory valuation engine (IN006/IN007)** — the highest-severity gap. `erpStockLedgerEntries.valuation_rate` is a raw stored number with no FIFO layer/queue or moving-average recompute logic behind it. Without this, COGS and balance-sheet inventory values are not trustworthy, which is fatal for a platform targeting CA firms.
2. **No procurement workflow above the PO (PR004, PR007, PR008)** — no RFQ, no purchase requisition, no approval routing. Firms can't run a compliant source-to-pay process; every PO is effectively a standalone document with no upstream authorization trail.
3. **No multi-UOM conversion (IN017)** — UOM is free text, breaking the common "buy in boxes, stock in units" pattern used by virtually every distribution/trading client.
4. **No batch/serial tracking (IN008/IN009)** — blocks any client with expiry-dated goods, warranty items, or regulatory lot-traceability needs.
5. **No 3-way match logic or vendor scorecarding (PR013/PR015)** — the tables link correctly (PO→Receipt→Invoice) but there's no tolerance-checking or automated vendor performance computation, both high-value, highly automatable features.

**Library candidates for hard narrow sub-problems (not full suites):**
- **FIFO/weighted-average valuation**: no mature narrow npm/GitHub library exists for this — most implementations are ERP-embedded (ERPNext's `stock_queue`, Odoo's `stock.valuation.layer`) rather than extractable packages. **Recommendation: build in-house**, modeled directly on ERPNext's FIFO-queue-per-item-warehouse pattern (simpler to port than Odoo's layer-consumption graph).
- **Barcode/QR generation**: `bwip-js` (MIT) generates Code128/EAN/QR server-side to PNG/SVG — directly usable for IN015 label printing, no reinvention needed.
- **UOM conversion modeling**: no dedicated package; it's a simple conversion-factor table (as in ERPNext's `uom_conversion_detail`) — trivial to implement directly in Drizzle schema plus a small conversion utility, not worth adopting external code for.

### Note on prioritization

Deprioritizing Manufacturing and Quality is correct given VERIDIAN's actual customer base — CA firms, legal practices, and consultants have no BOMs, work orders, or shop floors to manage, and building SAP PP/QM-equivalent depth (finite scheduling, MRP explosion, ISO 9001 CAPA workflows) would be pure surface area with zero attached revenue. ERPNext's own architecture reinforces this: `Non Conformance`/`Quality Action` and the routing/workstation/job-card stack are tightly coupled to a shop-floor data model (warehouses, stock entries, BOM explosion) that has no analogue in professional-services work.

The one place a lightweight subset could plausibly matter is a **document-assembly or deliverable-kitting analogue**: a CA firm "manufacturing" a compliance filing bundle (multiple sub-documents assembled per a fixed checklist, akin to a shallow one-level BOM) or a legal firm assembling a standard contract packet from clause components. That maps loosely to MFG001 (single-level BOM) and QC001/QC004 (checklist-based inspection before delivery). However, this is already better served by VERIDIAN's existing compliance-checklist and document-workflow primitives rather than importing MRP/QM concepts — so even this analogy doesn't justify borrowing manufacturing data models. There is no genuine case for building any part of M06/M07 today.

## Summary

The clearest, highest-priority gap is that **VERIDIAN has zero real payroll computation today** (PAY001–PAY002, PAY007): no salary structure engine, no batch payroll run, no payslip generation. The existing GRC "Payroll Compliance" module is a checklist/deadline tracker built in an earlier wave — it does not compute a single rupee of salary or tax, so this is a green-field build, not an extension.

Second, and arguably higher-risk: **Indian statutory payroll (PF/ESI/Professional Tax/TDS)** (PAY003–PAY006) is not a generic payroll problem — PF wage ceilings, ESI thresholds, state-wise PT slabs, and TDS regime rules change via periodic government notifications (EPFO/ESIC circulars, Finance Act amendments). Getting this wrong isn't a UX bug, it's a compliance and financial-liability event for customers (CA/legal firms whose entire value proposition is compliance). This must not be hand-rolled from scratch without a maintained rule source.

Third, **recruitment/ATS and performance appraisal are complete gaps** (HR003–HR005, HR010) — lower legal risk than payroll but high sales-relevance for HRMS positioning against ERPNext/Zoho.

**Concrete candidates to read/integrate:**
- `frappe/india-payroll` (GPLv3) — small, early-stage (12 stars) but has purpose-built `epf.py`, `esi.py`, `professional_tax.py`, `lwf.py`, `income_tax_utils.py`. Worth reading as a *reference implementation of the rules*, not production-ready as-is given its size/maturity — verify against current-year notifications before trusting any constant.
- `frappe/hrms` (GPLv3, ~8.2k stars) — mature India payroll logic embedded in `salary_slip.py`, `gratuity_rule_slab`, `income_tax_slab`; the most battle-tested open reference for structure.
- No robust, actively-maintained standalone npm package for Indian PF/ESI/PT/TDS calculation was found. **This needs to be built as an internally-owned, versioned rule engine** (not hardcoded constants) with a clear update process tied to government notifications — treat it as a compliance-grade component requiring legal/CA sign-off on rules, not a one-time dev task.

## Section 7: Document Management & Workflow/Approvals (M12, M13)

# M12 — Document Management & M13 — Workflow & Approvals

## Research Notes (grounding, not code copied)

**Frappe/ERPNext — Workflow engine** (confirmed via `frappe/frappe` develop branch, `frappe/workflow/doctype/`): this is a framework-level, not app-level, feature. The real doctype shape:
- **`Workflow`**: one row per workflow, linked to any `document_type` (any doctype), `is_active`, `send_email_alert`, `enable_action_confirmation`, plus a `workflow_state_field` (default `workflow_state`) added to the target doctype.
- **`Workflow Document State`** (child table): each state (`Draft`, `Pending Approval`, `Approved`...) maps to a `doc_status` (0=Draft/1=Submitted/2=Cancelled), an `allow_edit` Role, optional field-update side effects (`update_field`/`update_value`, with `evaluate_as_expression`), per-state email notification toggle.
- **`Workflow Transition`** (child table): `state` → `action` → `next_state`, gated by an `allowed` Role, an `allow_self_approval` flag, and a free-text Python `condition` (e.g. `doc.grand_total > 0`) — the amount-threshold/conditional-routing mechanism.
- **`Workflow Action Master`**: a named vocabulary of actions ("Approve", "Reject", "Submit for Review").
- **`Workflow Action`**: the actual per-document, per-user pending-action record — polymorphic via `reference_doctype` + `reference_name`, `status` (Open/Completed), `permitted_roles`, `completed_by`. This would drive an approvals inbox.

Net effect: one engine, attachable to *any* doctype, with role-gated multi-state transitions and conditional rules — precisely the reusable shape VERIDIAN lacks. It is not a parallel/multi-approver-per-step engine natively (each transition names one allowed Role, not a quorum) — a real limitation worth noting rather than over-claiming.

**Frappe — `File` doctype** (`frappe/core/doctype/file/file.json`, confirmed): polymorphic attachment via `attached_to_doctype` + `attached_to_name` + `attached_to_field`; folder hierarchy via self-referential `folder`/`is_folder`; `is_private`, `file_url`, `content_hash` (dedup/integrity), Dropbox/Google Drive sync flags. No native version-chain field on `File` itself — ERPNext relies on `track_changes` + a separate versioning doctype for full history; true version-controlled *documents* (vs. transactional attachments) are a known ERPNext gap versus SharePoint.

**Odoo `documents` app**: workspace/folder tree with tags and facets, activities/approval mixin (`mail.activity`) for review requests, versioning built on `ir.attachment` (checksum-based, new attachment record per revision rather than a first-class version object).

**VERIDIAN's actual current schema** (confirmed): a `documents` table today is scoped mainly to compliance items/notices (`complianceItemId`, `noticeId`, `fileUrl` into a private Supabase bucket) — not a general DMS. On workflow: there IS already a generic `approvalRequests` table (`entityType`/`entityId` polymorphic, `requestedById`/`approvedById`, single `pending→approved/rejected` enum) used for policy-publish/RPT maker-checker — real but minimal single-step approval, not multi-level/conditional. Separately, `pmsWorkflowTransitions` implements a genuinely ERPNext-like per-issue-type, per-role transition table — but it's hard-wired to PMS issues only, with no `Workflow`-style reusability across doctypes.

## Recommendation

Yes — build the shared **Approval Workflow Engine** as a dedicated wave, and prioritize it over Document Management. The evidence for leverage is concrete, not speculative: VERIDIAN already has *two* independent, non-reusable implementations of the same underlying problem (`approvalRequests` — generic but single-step maker-checker; `pmsWorkflowTransitions` — genuinely configurable multi-state but PMS-only), plus at least four more modules (GRC compliance items, ERP journal entries, invoices, purchase orders) each hand-rolling their own draft/submitted/cancelled enum with zero shared logic. That's six-plus call sites paying the cost of reinventing the same primitive, and the ERP expansion is about to add more. Frappe's own architecture is the existence proof: `Workflow`/`Workflow Transition`/`Workflow Document State` is a framework-level primitive precisely because dozens of ERPNext apps needed it — the exact shape VERIDIAN is heading toward.

The proposed schema — `approvalWorkflows` (definition), `approvalSteps` (ordered/parallel steps with role + condition), `approvalRequests` (generalized from the existing table, keeping its `entityType`/`entityId` polymorphism), plus a new `approvalStepInstances` or similar for per-step state — should absorb `pmsWorkflowTransitions`' conditional-transition logic and build directly on the two things VERIDIAN already does well: `hasRole` RBAC (for the "allowed role per step" gate, mirroring `Workflow Transition.allowed`) and the immutable `audit_logs`/`logActivity` pipeline (for WF006's audit trail, so no new logging plumbing is needed). This turns Workflow & Approvals from six scattered enums into one engine the ERP module can adopt from day one rather than becoming a seventh reinvention.

Two concrete external references worth evaluating (grounding only, not adoption without review):
- **`xstate`** (MIT-licensed, `npm i xstate`) — a mature, well-tested finite-state-machine/statechart library. Rather than hand-rolling transition validation and parallel-state logic in application code, `xstate` natively models parallel states (covers WF003's multi-approver-in-parallel gap that neither Odoo nor ERPNext fully solve) and guarded transitions (a typed analog to ERPNext's `condition` field), while remaining a pure library with no runtime lock-in — a good candidate for the state-transition core underneath `approvalSteps`.
- **`frappe/frappe`** itself (github.com/frappe/frappe, MIT) — not for code reuse, but as a living reference implementation to revisit during schema design, particularly the `Workflow Action` polymorphic-inbox pattern (`reference_doctype`/`reference_name` + per-user `Open`/`Completed` status), which maps almost directly onto what a VERIDIAN "My Approvals" inbox view would need to query.

## Summary

**Security & Roles (M14) is a genuine strength, not a gap.** Across role-based access control, audit logging, and API/session management, VERIDIAN meets or matches all four benchmarks. On two dimensions it goes further than any of them: row-level security is enforced as genuine PostgreSQL RLS at the database layer (`app_runtime_org_scoped` + `service_role_bypass` policies, verified after every one of 49+ schema waves), whereas Odoo's `ir.rule`, ERPNext's User Permissions, Zoho's workspace restrictions, and even SAP's authorization objects all enforce row security in application code, not the database itself — a materially stronger security boundary. Second, the combination of API-key auth, Purpose-Bound AI (restricting which AI tools can fire per business domain), and a deterministic Policy Enforcement Engine gating every LLM call against a written AI constitution has no equivalent in any of the four benchmarks — none of them were built with an AI-native threat model in mind. The two real gaps — two-factor authentication and IP allowlisting — are narrow, well-understood, bolt-on features rather than architectural weaknesses.

**Reporting & BI (M15) is the real gap, and it's not evenly distributed.** Ranked by priority:

1. **RPT001 — Standard financial reports.** This is the top priority because it is not a standalone BI gap — it's the same missing capability already flagged as the Finance module's #1 named gap (Trial Balance/P&L/Balance Sheet schema exists, zero report-generation service layer). Fixing this closes two gaps at once.
2. **RPT003 — Dashboard/KPI widgets.** Partners need at-a-glance visibility; currently nothing renders visually beyond Wave 38's raw threshold alerts.
3. **RPT002/RPT006 — Custom report builder and cross-module analytics.** Needed once basic reports exist, to let firms combine compliance, billing, and ops data without engineering effort.

Rather than building a BI engine or adopting a full platform, the pragmatic path is a lightweight embeddable charting library layered on the existing Next.js stack: **Tremor** (`@tremor/react`, MIT-licensed, built on Recharts, now maintained by Vercel — a natural fit given VERIDIAN's deployment target) for pre-styled KPI cards/dashboards, or **Recharts** directly (MIT, 2.4M weekly downloads) for lower-level composable charts if more customization is needed. Either avoids the cost of Power BI-class infrastructure while directly targeting RPT001 and RPT003.

## Summary
