## 2. Test design (as executed)

- 100 synthetic personas across 15 construction-company role archetypes, 5 tasks each (500 total), generated by GPT-OSS-120B.
- Originally specified to run entirely on Groq's free tier; **switched mid-run to Cerebras** (same GPT-OSS-120B model) after Groq's free tier turned out to have an undocumented-to-us 200,000 tokens/day cap that the persona-generation phase alone exhausted (§5, Incident 2). This does not affect the production cost numbers in §1/§3 — those are for the real `task_oa` layer, which independently uses Groq for its own floor tier and never hit this cap (only 482 calls, ~284K prompt + 179K completion tokens, comfortably under Groq's daily limit for that org's own key).
- Task execution ran through the real service layer (`createTask`/`task-execution-engine.ts`), not HTTP — same code path production traffic uses, bypassing only session-cookie auth (impractical for 100 synthetic users), per protocol §3.
- Budget guardrails ($3 Cerebras, $1 GLM-5.2), wall-clock cap (90 min), per-persona iteration cap (8), and rolling error-rate halt (30% over 50 tasks) were all live; none triggered.

## 4. Functional findings

### 4.1 Task execution — 99.8% success, self-healing

499/500 succeeded on first or retried attempt. The harness's `withRetry` wrapper (added after two earlier network-flake crashes, §5) absorbed transient Postgres connection errors without losing data.

### 4.2 The one failure — real, reproducible robustness gap

Task "KPIs Dashboard Refresh" failed with, verbatim (from `task_chat_messages`):
> `Execution failed: JSON Parse error: Unterminated string. You can retry by editing and resaving the task.`

Root cause: the LLM response for this task's plan/dispatch got truncated mid-JSON (same class of failure as the harness's own persona-gen truncation bug, §5 Incident 1 — GPT-OSS-120B is a reasoning model that spends real completion-token budget on hidden chain-of-thought before emitting the JSON answer, so any tight `max_tokens` risks cutting off the answer). Two real gaps this exposes:
1. **No automatic retry on JSON-parse failure** at the `task_oa` layer — the user is told to manually "edit and resave," which works (confirmed, see §4.4) but is a worse experience than a silent one-shot retry would be.
2. **Failed-before-model-call tasks don't get an `orchestra_executions` row**, so there's no `promptTokens`/`completionTokens`/cost record for the failed attempt — a blind spot for both debugging and cost accounting (the tokens for that failed attempt *were* spent, just not logged anywhere queryable).

### 4.3 Mode pills / capability tree — confirmed PROJEXA-scoped and working

Directly queried `buildCapabilityTree({ orgId })` for the demo org. Result: **3 top-level nodes**, all construction-relevant — `Product` (containing the seeded project's `status_update`/`log_task`/`flag_risk` leaves), `Construction Intelligence` (org-wide `list_delayed_activities`/`list_over_budget_projects`, plus per-project `Project Dashboard`/`Budget Status`/`KPI Status`/etc., each carrying a `fixedInputs.projectId` so selecting a mode pill passes the right project context straight through). No GST, compliance, or other-vertical nodes appeared. All 7 required `global`-tier construction worker agents (`get_construction_project_dashboard`, `get_construction_budget_status`, `get_construction_kpi_status`, `generate_construction_progress_summary`, `detect_construction_budget_schedule_risk`, `list_delayed_activities`, `list_over_budget_projects`) are present in production. This directly answers the original ask ("check if the dynamic mode pills + option selectors are... specific for PROJEXA and the respective modules") — **yes, confirmed working as designed.**

### 4.4 Task editing — confirmed working (via supplementary direct test)

The load-test harness generated ~10% of tasks phrased as edits (e.g. one sampled task was literally titled "Edit previous request") but **did not actually exercise the real edit code path** — `task.kind` was only used in error-log labeling, never routed to `updateTask()`. This is a harness gap, not a product gap (see §6). To close the gap without re-running the full test, 5 real tasks from the completed run were edited directly via `updateTask()`: all 5 succeeded, title and priority changes persisted correctly on re-read. **Editing works; the load test itself just didn't prove it at scale.**

### 4.5 Orchestra loops — only 1 of 5 is reachable by PROJEXA today

The original ask was to validate "all 5 orchestra loops working." The actual layer inventory:

| Layer | Status for PROJEXA |
|---|---|
| `task_oa` | **Live, real.** This is what this test exercised — all 499 logged executions are on this single layer, `eventType: task_execution.planning`. |
| `page_agent_oa` | Disabled. PROJEXA has no UI (only `/api/v1/projexa/*` routes exist), so a page-level agent has nothing to attach to. |
| `global_intelligence_oa` | Dormant — zero call sites in the codebase. Not wired to anything, PROJEXA or otherwise. |
| `meta_oa` | Platform-internal audit layer, not a per-task/per-user execution path — not something a construction-company user's task would ever route through. |
| `facilities_management_register_digitize_oa` | Belongs to a different product (FM register digitization), not PROJEXA. |

**This test could not validate 5 working orchestra loops for PROJEXA because only 1 of the 5 applies to PROJEXA at all in the current build.** This isn't a test-coverage gap so much as an architecture-maturity finding: if the intent is genuinely "5 loops all contributing to PROJEXA task quality," 4 of them need to either be built out for construction use cases or the "5 loops" framing needs to be revised to reflect what's actually live per-product. Recommend this go to z.ai as a scoping question, not a bug fix (see §7).

## 5. Incidents during setup/execution (chronological)

1. **`tasks.priority` column missing in production** — Wave 148's migration (`drizzle/0130_wave148_task_priority.sql`) existed in the repo but had never been applied to the live DB. This broke **all** task creation, not just this test. Fixed live via direct SQL before any test data was generated (this predates and is separate from the test itself, caught by the test's first `createTask` dry run).
2. **Harness `maxTokens: 120` too small for persona-gen JSON** — GPT-OSS-120B (reasoning model) spends real completion tokens on hidden reasoning before the JSON answer; 120 was insufficient, causing systematic truncation failures. Fixed by raising to 400 (persona gen) / 900 (task gen, after also tightening the prompt to ask for shorter text).
3. **Groq free-tier RPM (30) and TPM (8,000) limits** — hit during dry-runs; added an 8s/call throttle plus retry/backoff. Absorbed successfully in dry-runs.
4. **Groq free-tier TPD (200,000 tokens/day) — the harder limit.** Discovered live: the persona-generation phase alone (100 calls) consumed the *entire* daily allowance, because GPT-OSS-120B's hidden reasoning tokens count against every limit. No amount of RPM/TPM throttling fixes a *daily* cap already exhausted. **Switched harness generation calls to Cerebras** (same model, per the pre-approved tiered budget: free Groq → free/paid Cerebras). This is itself a real finding for the platform: **Groq's free tier alone cannot sustain even a single 100-user synthetic cohort's worth of generation traffic in a day** — worth knowing before assuming the floor tier is "free" at any real scale (see §7 recommendation).
5. **A background process was silently lost** during an environment reconnect mid-run (session-level event, unrelated to the test logic) — the underlying `bun` process had already died by the time it was noticed. Recovered by detecting the dead process via `ps`, cleaning up the partial org, and relaunching cleanly rather than attempting to resume in place.
6. **Drizzle's `sql` array-binding gotcha** — a cleanup script's `ANY(${array})` syntax produced `ANY(($1, $2))` (a row expression) instead of a real Postgres array, causing `op ANY/ALL (array) requires array on right side`. Fixed by switching to `IN (${sql.join(...)})`. Minor, harness-only, no production impact.

## 6. Harness gaps (fix before any future run at this scale)

- Task-edit and ambiguous-task *kinds* are generated but not actually routed to different code paths during execution (§4.4) — should call `updateTask()` for `kind: "edit"` entries to make editing a first-class part of automated runs, not a manual follow-up.
- Silent-failure paths (`status !== "failed"` check without a thrown error) don't get a `log()` call with detail — only the `catch` block does. Should log the `result`/`confirmed` object's failure detail too.
- No persistent mapping from generated task → its `kind` was kept once written to the DB, making later kind-specific analysis (like §4.4) require picking arbitrary tasks rather than exactly the "edit"-tagged ones. Should stash `kind` in `task.description` or a side table for future runs.

## 7. Recommendations for Z.ai GLM-5.2 (dispatch pending, see task tracker #12)

1. **Add one automatic retry on JSON-parse failure** in the `task_oa` execution path before surfacing "please edit and resave" to the user (§4.2) — cheap, high-value UX fix given the root cause (reasoning-model completion truncation) is itself intermittent and retry-shaped.
2. **Log failed-before-model-call attempts to `orchestra_executions`** (even with null token counts, or best-effort token counts from the truncated response) so cost/failure accounting doesn't have a blind spot (§4.2, §3.1).
3. **Scoping decision needed, not a code fix:** clarify whether `page_agent_oa`, `global_intelligence_oa`, and `meta_oa` are meant to eventually serve PROJEXA, and if so what "PROJEXA task passes through this layer" would even mean given PROJEXA has no UI yet (§4.5). This affects whether future load tests should treat "5 loops" as the right success bar.
4. **Document Groq's TPD=200,000/day ceiling** in the floor-tier architecture notes (`orchestra-model-resolver.ts` or `AGENTS.md`) so nobody re-discovers this by burning a day's quota again (§5 Incident 4) — and consider whether the platform-default floor tier needs a same-day Groq→Cerebras failover for *production* traffic, not just this test harness, if daily volume could plausibly approach that cap.
