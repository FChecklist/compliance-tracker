# VERIDIAN AI OS — Deep Technical Audit

**Date:** 2026-07-09 | **Scope:** compliance-tracker repo (VERIDIAN AI OS + PROJEXA), ~460 tables (357 confirmed live), 114 services, 573+ API routes, 145+ pages, ~61K LOC, ~130 build "waves".

**Methodology, stated plainly:** 9 parallel research agents each owned a cluster of the sections below, going deep where risk concentrates (auth, RLS, money-handling ERP code, AI dispatch safety) and sampling representatively elsewhere. Several findings were verified against the **live production database** via Supabase MCP (`get_advisors`, direct SQL) — these are marked 🔵 and are authoritative. Code-only findings are marked 🟡. This is not a claim of having read all 573 routes individually — see each section's own sampling note. Every finding cites real file:line evidence.

**Format:** every finding uses **[SEVERITY] Title** → Why it matters → Recommended solution → Estimated effort → AI-automatable (Yes/No/Partial).

## 1. Executive Summary

VERIDIAN AI OS is a genuinely large, largely real, production-deployed system — not a prototype. Multi-tenant RLS isolation is the single strongest-verified guarantee in the codebase (independently re-proven with live switched-role tests across dozens of waves). The architecture is unusually well-documented for AI-agent-driven development: governance docs grade every claim `[ENFORCED]`/`[POLICY ONLY]`/`[NOT APPLICABLE YET]` against real file:line citations, and 130 waves of changelog show real bugs found and fixed rather than hidden.

That said, this audit found **two Critical findings that change the honest answer to "is this an AI OS":**

1. **The worker-agent execution engine does not execute worker agents.** Outside ~20 hardcoded dispatch branches (compliance, GST, construction), an approved/published worker agent can be proposed and published but the engine only records a plan against it — it never runs. This is self-documented in the code's own comments, not hidden, but it means the core "propose an agent, it does the work" value proposition doesn't exist yet for the vast majority of the registry.
2. **The CA/firm multi-tenant gap is a live access-control bug, not a missing UI screen.** The Firm practice module's ~25 routes never pass `clientIds` into the tenant-scoping context — a staffer restricted to one client can query another client's full data through that module today. This is the concrete root cause behind an independent CA evaluator's 5.5/10 rating.

Beyond those two, a third Critical, externally-reachable finding: an **MCP write tool (`create_compliance_item`) accepts foreign-key arguments with no cross-tenant validation** — any write-scoped API key holder can create a compliance item referencing another org's department/user.

The database itself is well-run for its scale (357 live tables, 100% cuid2 consistency, 100% RLS-enabled) but has two structural weaknesses: **foreign-key integrity is ~99% unenforced at the DB level** (11 of 972 FK-shaped columns have a real constraint) and **zero explicit indexes exist in `schema.ts`**, confirmed live as a 203-unindexed-FK backlog across 136 tables. Neither is a security hole; both are real correctness/performance risk as data volume grows.

No payment processor exists despite the public pricing page claiming Razorpay processing — a real, currently-false commercial statement that should be fixed immediately regardless of any other roadmap priority.

**What's genuinely good:** RLS enforcement, the AI safety "belt and suspenders" pattern where it's wired, module-reuse discipline across ERP/PMS/PROJEXA (near-identical scaffolding despite no shared generator), a real 6-workflow CI pipeline, near-zero lint-suppression debt, and an unusually honest documentation culture that graded its own gaps rather than hiding them.

## 2. Overall Architecture Review

One tenant, one AI Orchestra (5 layers, 3 of 5 now live — `task_oa`, `user_assistant_oa`, `meta_oa`; `customer_account_oa` and `global_intelligence_oa` remain dormant, `global_intelligence_oa` with zero references anywhere in the codebase). Product branches (19 named verticals, 3 tiers: repackage/moderate_build/ground_up) reuse one multi-tenant hierarchy and one RLS pattern by architectural mandate — this discipline has genuinely held (ERP/PMS/PROJEXA are structurally near-identical despite being built independently, confirmed by direct comparison).

**[HIGH] The product-branch catalog (`product_branches.status`) is confirmed stale live** — Construction shows `'planned'` and Facilities Management shows `'building'` despite both having substantial real, shipped code (PROJEXA: 15 tables/10 services/~55 routes; FM: 13 tables/8 services + a live public page). This is the third documented instance of this catalog drifting from reality (also flagged for these same two branches in `MASTER_AI_OS_ARCHITECTURE.md` itself, written earlier and already known-stale). *Why it matters:* if this table drives any user-facing "coming soon" messaging or AI-agent decision-making, it's actively misrepresenting two live product lines. *Recommended solution:* a wave-closing checklist step that updates `status`/`build_tier` as part of "done" criteria. *Effort:* Small (data fix) + Small (process). *AI-automatable:* Yes for data, Partial for process.

## 3. Repository Structure Review

`src/lib/db/schema.ts` (7,341 lines, 356 tables — corrects the estimate this audit was briefed with of ~460) is the single largest structural risk point: it has zero index declarations (all indexing lives in separate raw SQL migrations, invisible from the file that's supposed to describe the schema) and is imported by 227+ files, meaning nearly every serverless function cold-starts the full 460-table relation graph regardless of what it actually touches.

**[LOW] `CLAUDE.md` is badly stale** — states "Drizzle schema (9 tables, 6 enums)," off by ~350 tables (a leftover from the pre-VERIDIAN "compliance-tracker" origin), and claims shared `DataTable`/`StatusBadge`/`DashboardCard` components exist in `src/components/` when they don't (confirmed: 21+ pages independently hand-roll their own status-color maps instead). *Recommended solution:* correct both claims; either build the claimed shared components or stop claiming them. *Effort:* 10 min (doc fix) + 1-2 days (component extraction, optional). *AI-automatable:* Yes.

See `docs/master/MODULE_MAP.md` for the full domain→table→service→route→page inventory.

## 4. Authentication & Authorization

Two-tier auth model (`requireAuth()` session-only vs. `requireAuthOrApiKey()` for API-key-capable domains) is applied **consistently**, not inconsistently, within each tier across the ~15 routes sampled — a positive finding.

**[MEDIUM] No role-change or off-cycle deactivation path exists for users.** `src/app/api/users/route.ts` has only `GET`/`POST` — no `PATCH`/`DELETE` on `/api/users/[id]` anywhere. A user's role is fixed at invite time forever; the only deactivation path is the Access Review module's periodic certification cycle, not an emergency same-day action. *Recommended solution:* add `PATCH /api/users/[id]` (role change) and a deactivate action, both `admin`+-gated, mirroring the existing org-check pattern in `clients/[id]/access/route.ts`. *Effort:* Small (half a day, pattern already exists twice to copy). *AI-automatable:* Yes.

**[LOW] No true "forgot password" flow** — only a "send magic link instead" button; re-verified as still true (`functional_testing.md`, 2026-06-30). *Effort:* 15 min (relabel) to 2-4 hours (real reset flow). *AI-automatable:* Yes.

**[LOW] No "remember me" option** on login. Re-verified as still true. *Effort:* 30 min-4 hours. *AI-automatable:* Yes.

Two prior QA findings are **now confirmed fixed**: the pricing comparison page (now a full 18-row feature table) and the hardcoded org-name bug (org name is fetched live from `/api/me` everywhere).

## 5. Multi-tenancy

**[CRITICAL] The Firm practice module never applies per-client access scoping despite the schema fully supporting it.** All ~25 `src/app/api/the-firm/**` routes call `withTenantContext({ orgId })` — never `{ orgId, clientIds }`. Repo-wide, `clientIds` is only actually passed in 2 call sites (`worker-agent-service.ts`, `worker-agents/route.ts`) — neither is the Firm module. `userClientAccess.accessLevel` (`'full'|'aggregate_only'`) exists in schema and the `withTenantContext`/GUC plumbing (`app.current_client_ids`) is fully built — it's simply never populated by this module's routes. **Concretely: a junior associate scoped to Client A's `aggregate_only` access can call `/api/the-firm/clients/[clientB]/invoices` and get Client B's full data**, because the route only checks `orgId`.

This is the code-level root cause of the independent CA evaluator's 5.5/10 rating — not "a missing UI flow" as it might appear from the schema/UI alone, but a genuine, present-day authorization gap.

- *Recommended solution:* In every `the-firm/*` route, after `requireAuth()`, resolve the caller's `userClientAccess` rows into a `clientIds` array and pass it into `withTenantContext`. Add RLS policies on firm-scoped tables (`firm_engagements`, `firm_invoices`, etc.) checking `client_id = ANY(compliance.current_client_ids())`, falling back to full org access only for roles with no `userClientAccess` rows (partners/admins).
- *Example architecture:* a `resolveClientScope(userId, orgId)` helper returning `{clientIds?: string[]}`, mirroring the existing `resolveCompanyScope` pattern already proven in `erp-financial-report-service.ts:246`.
- *Effort:* 3-5 days (RLS policy work + ~25 route updates + regression testing).
- *AI-automatable:* Partial — route updates are mechanical; RLS policy design needs human review.

**[HIGH] The broader multi-client UX is also shallow at the product-surface level.** `/clients` page is a card grid with a raw `window.prompt()` for adding a client — no client detail page (cards don't navigate anywhere), no UI for legal entities despite a working API, no UI for granting/viewing per-user client access despite a working API, and **no client-context switcher anywhere in the product** (grepped every component for `clientId`/`activeClient`/`selectedClient` — zero matches). Most of the ~460 tables outside the `clients` hierarchy itself have no client-scoping dimension at all. *Recommended solution:* this needs a genuine product decision — pervasive client-scoping (à la QuickBooks Online Accountant) vs. keeping `clients` a lightweight contact list. *Effort:* Large, multi-week, phased. *AI-automatable:* No for the decision; Partial for execution once scoped.

RLS itself (the tenant-isolation mechanism, distinct from the client-sub-scoping gap above) is genuinely solid: `app_runtime` role correctly lacks bypass/superuser privileges, and the sampled API routes across compliance/ERP/PMS/construction/SSO consistently delegate to `withTenantContext`-scoped services rather than hand-rolled `WHERE org_id=` checks.

## 7. Security Assessment

**[CRITICAL] MCP write tool `create_compliance_item` has no cross-tenant referential validation.** `src/app/api/mcp/route.ts` (~line 423-440): the handler correctly scopes `org_id` from the caller's API key, but takes `department_id`/`assigned_to_id` directly from the external MCP client's arguments with **no check they belong to the same org**. Externally reachable by any customer holding a write-scoped `vk_...` key. *Recommended solution:* add an `assertBelongsToOrg(sb, table, id, orgId)` helper, call it for every foreign-key argument before insert. *Effort:* Small (1-2 hours). *AI-automatable:* Yes.

**[HIGH] Policy Enforcement Engine coverage has not grown — and the gap has widened.** Still exactly 3 wired call sites (VERI Chat, VERI FDE, Page Agent proxy) out of 13 real LLM call sites identified. 7+ newer surfaces (construction AI, CRM AI, meeting intelligence, FM register digitization, visitor/sales AI, `document-extraction-service.ts`, the task-planning call itself) have **zero** prompt-injection/personal-use/domain-validity gate — built without wiring in the existing, free, zero-latency check. *Recommended solution:* treat `enforcePolicy()` as a required import for any file calling `callLLM*`; enforce via a lint rule or CI grep check; retrofit the 7+ services. *Effort:* Medium (~1-2 days total). *AI-automatable:* Yes.

**[MEDIUM] Middleware route allowlist has drifted a 4th time.** `/connectors`, `/gst-reconciliation`, `/tds-returns`, `/the-firm-practice` all exist as real pages but are absent from `PROTECTED_APP_ROUTE_PREFIXES` — unauthenticated visitors get a 200 page shell instead of a redirect. Same class of bug fixed 3 times before (34/51 missing, then `/erp`, then `/knowledge-base`+`/automation`). No data leaks (API routes are independently `requireAuth()`-gated), but the recurrence pattern is the real finding. *Recommended solution:* stop hand-maintaining a duplicate list — generate the allowlist at build time from the real `(app)/` directory listing, or invert to "protect everything except an explicit public allowlist" (safer failure direction). *Effort:* Small (immediate fix) to half a day (generator + CI drift check). *AI-automatable:* Yes.

**[MEDIUM] 🔵 Migration files are not a trustworthy record of the live security posture.** `users`, `departments`, `compliance_items` have live RLS+FORCE+2 policies each with **zero corresponding SQL in any of 115 migration files** — applied out-of-band at some point (manual `db:push` or an untracked baseline). A static-file-only audit would have drawn wrong conclusions in either direction. *Recommended solution:* reconcile live schema against migration history (`drizzle-kit pull`/introspection), commit a catch-up migration, add a CI drift-detection gate. *Effort:* Medium. *AI-automatable:* Partial.

**[MEDIUM] 🔵 `FORCE ROW LEVEL SECURITY` is enabled on 0 of 357 tables.** Currently non-exploitable — `app_runtime` (the actual app connection role) is confirmed `rolbypassrls=false` and is not the table owner (`postgres` owns everything) — but fragile: any future `ALTER TABLE ... OWNER TO app_runtime` would silently disable RLS for that role on every affected table with no error. *Recommended solution:* one migration adding `FORCE ROW LEVEL SECURITY` to all 357 tables (free, zero behavior change today); add a CI check requiring it alongside `ENABLE` for new tables. *Effort:* Small. *AI-automatable:* Yes.

**[LOW] No rate limiting on session-authenticated routes or the 18 AI-backed API routes.** 568 of 573 routes have zero application-level throttle; a compromised session or a runaway script could hammer any mutating/AI-backed endpoint with no cap, risking both cost blowout and pooler contention. *Recommended solution:* a lightweight per-user/org rate limiter (e.g. `@upstash/ratelimit`, already Vercel-deployed) on the highest-risk routes first (user invite, SSO config, API key creation, `/api/ai/*`). *Effort:* Medium (1-2 days). *AI-automatable:* Yes for mechanical parts.

**[LOW] 🔵 Standard Supabase hardening gaps** (from live `get_advisors`): 3 `SECURITY DEFINER` views (`ai_export_*`) run with creator privileges, not caller's; `compliance.conversation_org_id()` is a `SECURITY DEFINER` function executable by `anon`+`authenticated` via RPC (not confirmed exploitable — body not reviewed); 8 functions have mutable `search_path` (classic hijack risk pattern); `hstore` installed in `public` not a dedicated extensions schema; leaked-password protection disabled in Supabase Auth. *Effort:* Small (a few hours total). *AI-automatable:* Partial.

## 8. Database & Schema Review

**Verified 🔵 live:** 357 tables (not ~460), 100% RLS-enabled, 100% cuid2 PK consistency, zero "declared but never migrated" drift (the historical pattern is fixed) — but one reverse case found: **`firm_client_portal_links` exists live with RLS+policies and zero trace in `schema.ts` or any migration file** — either created ad hoc outside the pipeline or an orphaned remnant. *Recommended solution:* investigate via deployment history; drop or retroactively wire up. *Effort:* Small. *AI-automatable:* Partial.

**[HIGH] Foreign-key integrity is ~99% unenforced.** 🟡 Of 972 `_id`-shaped columns, only **11** carry a real `.references()` — all clustered in one Wave 56/57 payroll/inventory band. Every other domain (GRC, PMS, CRM, construction, FM, GST, HR, worker-agents, sales) relies entirely on Drizzle's non-enforcing `relations()` helper plus service-layer discipline. Concrete unenforced examples span the general ledger itself (`erpJournalEntryLines.accountId`), invoices, `pmsIssues.projectId`/`statusId`, and `contractComplianceItems.vendorName` (not even FK-shaped — a free-text vendor name with no link at all). *Recommended solution:* triage by blast radius — money/compliance-critical tables first (ERP financials, `complianceItems`, `auditFindings`, `risks`), add `.references()` via `ADD CONSTRAINT ... NOT VALID` + `VALIDATE CONSTRAINT` to avoid locking. *Effort:* Large, multi-wave. *AI-automatable:* Partial (mechanical generation; `onDelete` semantics need human sign-off).

**[HIGH] 🔵 Zero explicit indexes in `schema.ts`; 203 unindexed FKs confirmed live across 136 tables** (`get_advisors(performance)`), concentrated on tables designed with `org_id` as the mandatory RLS filter predicate on every query. Plus 4 confirmed duplicate indexes (free wins to drop) and 260 unused indexes (including 58 ERP indexes that were correctly created at migration time but have **never been hit by a query** — direct evidence the "10 demo companies" multi-tenant test never generated real traffic). *Recommended solution:* clear the backlog in 2-3 domain-grouped batches (ERP procurement first); adopt `get_advisors(performance)` as a standing pre-release gate, not an ad hoc check. *Effort:* Medium. *AI-automatable:* Yes.

**[MEDIUM] One migration numbering collision:** `0101_gst_worker_agents.sql` and `0101_wave115_construction_boq_progress_diary.sql` share a number; ordering is currently harmless (disjoint tables) but is luck, not design. *Effort:* Small. *AI-automatable:* Yes.

**[MEDIUM] ~7 undocumented denormalized total columns** (`grandTotal`/`totalAmount`/`outstandingAmount` on POs/SOs/quotations/credit notes/invoices duplicate child-line sums with no comment) — inconsistent with this codebase's own normal practice of flagging such caches (e.g. `complianceCosts.amountPaid` is documented). *Effort:* Small (add comments) to Medium (add sync triggers/checks). *AI-automatable:* Yes.

**[MEDIUM] RLS policy naming has drifted on 62 of 357 tables** 🔵 — no privilege-escalation risk found (no policy mixes roles incorrectly), but ~44 tables use abbreviated non-standard policy names, making systematic audit harder (confirmed: took 3 separate queries to characterize correctly). 16 tables intentionally have only a `service_role`-only policy (plausibly correct, unverified per-table); 8 use a deliberate, more granular 5-policy split. *Effort:* Small. *AI-automatable:* Yes.

**[LOW] The migration-amendment-in-place practice is a real process risk as the team scales beyond one AI session with full context.** Several past migrations were edited after being applied ("folded into history for accuracy") rather than corrected via a new migration — safe for a single agent with full memory of what's live, unsafe for any second environment/operator replaying from file 0. *Recommended solution:* stop amending applied migrations; ship corrective migrations instead; add a CI checksum-drift guard. *Effort:* Small (process change). *AI-automatable:* No (policy), Yes (CI guard).

See §10 for data-model-specific redundancy findings (contactSubmissions/forgeProjectRequests, pmsTimeEntries/firmTimeEntries).

## 10. Data Model Improvements

**[LOW] Confirmed redundant table pair: `contactSubmissions`/`forgeProjectRequests`.** Column-for-column near-identical (draft→email-confirm-token→submitted lifecycle, `visitorId`-keyed, no `orgId`), differing only in payload shape. The code's own comment acknowledges the shared posture but doesn't justify the duplication. *Recommended solution:* either merge into one `landing_intakes` table with a `kind` discriminator + `payload jsonb`, or (lower-risk) extract a shared `intake-lifecycle-service.ts` both tables' routes call for the confirm/submit transitions. *Effort:* Medium (merge) / Small (shared service). *AI-automatable:* Yes.

**[LOW] Second, newly-confirmed redundant pair: `pmsTimeEntries`+`pmsBillableRates` vs. `firmTimeEntries`+`firmBillableRates`.** Same "timesheet + resolvable billable rate via nullable-userId org-default fallback" mechanic implemented twice for two consuming modules — `firmBillableRates`'s own comment explicitly says it "generalizes pmsBillableRates' existing convention," i.e., the duplication was a conscious choice, not an oversight. *Recommended solution:* lower priority than #1 above since both are actively used by genuinely different modules (PM issues vs. client billing) — worth a shared base table/service if a third billing-timer consumer ever appears; not urgent otherwise. *Effort:* Medium if pursued. *AI-automatable:* Yes.

**[LOW] `subscription_plans` vs. `erp_subscription_plans`/`erp_subscriptions` — verified NOT a duplicate**, flagged only so future audits don't re-investigate: the former is VERIDIAN's own SaaS pricing catalog, the latter is an ERP feature letting a *tenant* bill *their own* customers. A one-line cross-reference comment at each definition would save future audit cycles. *Effort:* 10 min.

See §8 for the FK-integrity and indexing findings, which are the higher-severity data-model issues overall.

## 12. UX/UI Review

## 13. API Design Review

The API layer is **the strongest part of this codebase** — a consistent `requireAuth()`/`requireAuthOrApiKey() → try/catch → ServiceError` convention holds from the oldest domain (compliance) to the newest (PROJEXA), a real achievement at 573 routes.

**[MEDIUM] Zod schemas in `src/lib/schemas/*.ts` are OpenAPI-documentation-only, not runtime request validation** — confirmed 0 of 573 routes import them for validation; only `openapi/generate.ts` uses them to produce the spec. Real validation is hand-written per service, independently of the schema, risking silent drift between what the OpenAPI doc promises and what the service actually accepts. *Recommended solution:* wire the existing schemas into their ~15-20 associated routes as real `safeParse` validation. *Effort:* 1 day for existing schemas; multi-week if extended codebase-wide (likely not worth it wholesale). *AI-automatable:* Yes for wiring existing schemas.

**[MEDIUM] `/api/v1/openapi.json` covers a minority of the real surface** — ~45-50 operations documented (compliance/tasks/notices/construction/some ERP+PMS) vs. ~90 real domains; HR, CRM, board governance, litigation, IP portfolio, cap-table, and dozens more have no `/api/v1` presence despite being fully built internally. Honestly self-disclosed in code comments. *Recommended solution:* treat `/api/v1` expansion as an explicit, prioritized backlog (CRM/HR likely first, given they're common integration targets) rather than ad hoc per-wave. *Effort:* ~2-4 hours per domain × 30 domains. *AI-automatable:* Yes, highly mechanical given the established pattern.

## 23. Performance Bottlenecks

**[HIGH] Live financial aggregations have a genuine, unaddressed performance cliff.** `accountBalancesInRange` (the shared core of every financial report) does a 3-table join filtered by `orgId`/`status`/`postingDate` — but `erp_journal_entries` has **no index on `posting_date` or `status`** anywhere in 115 migrations, despite every financial report filtering on exactly that combination. Trial Balance/Balance Sheet scan "from inception" with no lower bound, so cost grows forever, never just the reporting period. Separately, `listSupplierScorecards` is a confirmed N+1 — loops over every supplier serially (not `Promise.all`), 1+3 queries each, no `LIMIT` — ~800 sequential round-trips for a 200-supplier company. *Recommended solution:* (1) composite index `erp_journal_entries(org_id, status, posting_date)`; (2) materialize opening balances at period-close (`erpAccountingPeriods.status='closed'` already exists) so scans reset at each closed boundary instead of growing unboundedly, preserving the "always live, never duplicated" philosophy for the *current* period only; (3) rewrite the scorecard as one grouped SQL query. *Effort:* Index: 1 hour. Scorecard: half a day. Period-snapshot: 3-4 days (needs accounting-correctness sign-off). *AI-automatable:* Yes for index/scorecard; Partial for the snapshot design.

**[MEDIUM] Two ad hoc `postgres()` clients exist beyond the two designed connection pools, one with no `max` cap** — `embeddings.ts` and `ai-config-crypto.ts` each construct their own raw client (defaulting to postgres.js's 10-connection cap), alongside `db/index.ts` (max 1) and `tenant-scoped.ts` (max 5). A single request touching all four code paths could open up to 26 connections; under concurrent load this exhausts Supavisor pooler headroom well before its nominal capacity. *Recommended solution:* consolidate to one shared client per role, explicitly cap the two ad hoc clients at 1-2 given their low-frequency use. *Effort:* Half a day. *AI-automatable:* Yes.

**[MEDIUM] One of four hardcoded pooler-region fallbacks was never fixed after the documented Wave 45 region incident.** `ai-config-crypto.ts`'s `getConnectionString()` still returns the deleted MeetTrack project's region (`ap-northeast-2`) instead of the correct `ap-south-1` — dormant today only because `DATABASE_URL` stays set in Vercel, but a live landmine identical in shape to a real past outage, proving the original fix was applied file-by-file instead of via one shared function. *Recommended solution:* extract `getConnectionString()` into one shared module all 4 files import. *Effort:* 1-2 hours. *AI-automatable:* Yes.

**[MEDIUM] Compute/data-plane cross-region hop on every request** — Vercel deployment region `sin1` (Singapore) vs. Supabase project region `ap-south-1` (Mumbai), paying 30-60ms RTT on top of `withTenantContext`'s own per-request transaction overhead. *Recommended solution:* move Vercel deployment region to a Mumbai-area option if available (`bom1`), or explicitly document as an accepted tradeoff. *Effort:* Trivial config change + latency verification. *AI-automatable:* Yes for the change.

A real procurement blocker for any EU entity or DPDP-covered Indian company. *Recommended solution:* an async, admin-gated bulk export job for the top 15-20 customer-facing tables first. *Effort:* 1-2 weeks for a first cut. *AI-automatable:* Yes, mostly mechanical.

See §5 for the CA/multi-client gap (both the Critical access-control bug and the shallower UI-layer finding), §17 for the worker-agent-execution Critical finding, and §26/§27 for the E2E/APM Missing gaps that also belong conceptually here.

**[MEDIUM] No customer-facing status page or uptime history** — `/api/health` exists as basic liveness only. *Effort:* 2-3 days (a third-party status page is the pragmatic option given no in-house ops signal exists). *AI-automatable:* Yes.

**[Positive, corrects an open question] VERI Reward has a genuinely complete, polished UI** (314-line `/rewards` page: points balance, streaks, achievements with progress bars, referral links, team leaderboard, activity feed) — not backend-only as was previously unclear. ## 34. Module-by-Module: Construction ERP (PROJEXA)

**Read in full:** all 10 `construction-*-service.ts` files, `projexa-aliases.ts`. **Sampled:** ERP integration points, full directory listings.

**Overall:** Real schema+services+API layer, actively the most recently-built area of the codebase (waves in the 120s-130s). **Zero frontend pages exist** (§12). **Confirmed: PROJEXA is a partial island relative to core Accounting ERP** — it correctly *reads* from ERP (`erpSalesInvoices` for revenue, `erpBudgets`/`erpCostCenters` for budget, `erpSuppliers` aliased as "vendors" via `projexa-aliases.ts`, not duplicated) but its own cost-tracking writes (`constructionExpenseEntries`, `constructionAttendance.dailyCost`) **never post a journal entry** — by the expense service's own header comment, "never the source of truth for the underlying transaction." A construction project's real costs never reach Trial Balance/P&L/Balance Sheet; Finance and Construction would report different numbers for the same project.

**[HIGH] Expenses/labour costs never post to the GL** (detailed above). *Recommended solution:* auto-post (or let a user post) a JE debiting the project cost-center's expense account and crediting cash/payable when an expense/attendance-cost is recorded, linked via the existing `referenceType`/`referenceId` polymorphic pattern already used by `erp-cash-service.ts`; add `project_id` to `erp_purchase_invoices` to close a related vendor-cost report gap. *Effort:* Medium (schema addition + wiring, 1 wave). *AI-automatable:* Partial.

**[MEDIUM] Payroll's `percentage_of_gross` earning-component calculation silently computes as zero** (a bug in `erp-payroll-service.ts` shared with ERP proper, not PROJEXA-specific, but discovered during this pass) — a promised "second pass" once gross is known is never actually implemented; any such component is inserted as a `0`-amount payslip line with no error. *Effort:* Small (a few hours). *AI-automatable:* Yes.

**[MEDIUM] Professional Tax slab lookup silently falls back to the highest configured slab** if none matches, rather than surfacing a configuration gap — inconsistent with the codebase's own TDS handling 30 lines below, which correctly labels unconfigured tax as "not auto-calculated." *Effort:* Small. *AI-automatable:* Yes.

**[MEDIUM] Barcode/QR generation remains an open, previously-deferred gap** — confirmed still absent via repo-wide grep. *Effort:* Small-Medium. *AI-automatable:* Yes.

**[Positive] "Never guess" discipline holds through the most recent waves** — `erp-invoicing-service.ts`/`erp-accounting-service.ts`/`erp-payroll-service.ts` all still require explicit accounts/rates rather than defaulting, no erosion found in the 68-132 wave range.
