# 100% Completion Plan

**Built 2026-07-11**, in direct response to feedback that closing gaps found in `01-COMPARISON.md` requires an actual execution plan, not just a tree of findings. Covers every one of Tree 4's 149 sub-branches at full granularity, with narrow instructions, input, output, and guardrails per item -- not just the 11 headline items `30-gap-backlog.yaml` originally scoped.

## Files

- `01-phase0-owner-decisions.yaml` -- **6 decisions that must be made before building starts.** Building ahead of these risks real wasted work (e.g. building 150 named auditor roles, or a richer approval-option UI, that the Owner didn't actually want). Each carries a recommendation, not just an open question.
- `02-plan-items.yaml` -- **59 plan items across 4 phases**, the actual execution plan. Phase 1 = cheap verification (15 items -- several may turn out to already be done). Phase 2 = foundational builds that unblock later work (7 items). Phase 3 = independent feature builds (27 items). Phase 4 = large, separately-scoped initiatives measured in weeks not days (10 items, including 2 explicitly conditional on Phase 0 decisions).
- `03-coverage-matrix.yaml` -- **mechanical proof of completeness.** Generated by cross-referencing the plan items against Tree 4's 149 sub-branches, not hand-maintained. Confirms 0 sub-branches are uncovered: every one has either a plan item that closes it, or an explicit, cited reason no action is needed (already implemented, a Tree-1-internal cross-reference, non-code framing text, or formally superseded pending a Phase-0 decision).

## How "100% completion" is defined and verified here

Every sub-branch in Tree 4 falls into exactly one of 3 buckets, and the coverage matrix proves this mechanically rather than asserting it:

1. **Already complete** (22 sub-branches: `implemented`/`mostly_implemented`/`fully_implemented`) -- no plan item needed, confirmed by Tree 4's own audited status.
2. **No code action required** (18 sub-branches: `not_applicable_to_code` framing/vision text, or `duplicate_ref` cross-references Tree 1 itself pointed elsewhere) -- explicitly listed with the reason, not silently dropped.
3. **Needs a plan item** (109 sub-branches) -- every one is named in exactly one (or more, where genuinely shared) of the 59 plan items' `covers_subbranches` list.

This was not correct on the first attempt -- a mechanical cross-check after the initial draft found 13 sub-branches with a real, non-trivial status that had been left off every plan item's coverage list. Those 13 were individually reviewed and assigned (11 folded into the most topically-relevant existing plan item, 2 given an explicit "no action, formally superseded pending DEC-04" disposition) before this plan was considered done. The coverage matrix is what makes this auditable going forward, not just a one-time claim.

## Sequencing logic

Phase 1 (verification) comes first because several "gaps" may turn out to already be built -- verifying is cheaper than building, and building something that already exists wastes the exact effort this plan exists to direct well. Phase 2 (foundational) comes next because a meaningful fraction of Phase 3/4's items explicitly depend on it (e.g. every guardrail-related Phase-3 item depends on PLAN-16's leaf-registration work; PLAN-21's state-model work blocks PLAN-22/25). Phase 4 is last and explicitly flagged as large/separately-scoped -- these are not smaller versions of the same kind of work as Phases 1-3, and treating them as equally quick would misrepresent the real effort involved.

## What "100%" does NOT mean here

- It does not mean every plan item should be executed blindly. 2 items (`PLAN-58`, `PLAN-59`) are explicitly conditional on Owner decisions that could result in "no build needed" as the correct outcome (e.g. if DEC-03 ratifies the current architecture, `PLAN-59`'s Work Object unification retrofit should NOT be built -- that would be closing a gap the Owner decided wasn't actually a gap).
- It does not mean this plan itself is guaranteed complete forever -- if new sub-branches are added to Tree 1/Tree 4 later, the coverage matrix must be regenerated, not assumed to still be accurate.
