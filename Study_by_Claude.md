### 1. Vision — VERIDIAN as Cognitive AI Operating System (source lines 14–24)
- **Understanding:** VERIDIAN is explicitly defined by negation — not an ERP, PM tool, accounting system, construction platform, or CRM, and not a collection of apps — but a Cognitive AI Operating System (CAIOS) under which every application, module, feature, workflow, agent, API, integration, dashboard, report, and customer/industry solution exists as a governed component. No component exists independently.
- **Architecture/Schema implications:** Implies a strict containment model: every deployable unit must be traceable to a parent OS-level registry entry (this is elaborated in principles 16 and the Asset Registry). It argues against any module having its own persistence/auth/intelligence stack outside the platform's.
- **Gap vs. current repo:** Partially realized. The repo genuinely does host multiple "products" (VERIDIAN core + PROJEXA construction ERP) under one Next.js/Drizzle/Supabase stack with shared auth (`requireAuth()`), shared schema, and a shared `ai-os/` governance directory — this is real evidence of the "no component exists independently" principle being followed structurally. However, there is no single registry file/table that enumerates "every application, module, feature, workflow, worker agent..." as claimed — see principle 16 (Asset Registry) below, which is the more specific, checkable claim.
- **Implementation recommendation:** No new work implied beyond what's captured under principles 5/6/16 below; this section is the philosophical framing those principles operationalize.

### 2. Core Philosophy — Ownership Split (source lines 26–36)
- **Understanding:** Assigns ownership cleanly: the AI OS owns intelligence; applications own business functionality; modules own reusable capabilities; projects configure capabilities; customers consume; worker agents execute; third-party software extends; the Enterprise Cognitive Graph stores enterprise knowledge. Nothing should duplicate intelligence — everything should inherit it from the OS.
- **Architecture/Schema implications:** This is the clearest statement in the whole document of a required layering: Intelligence (OS) → Capability (Module) → Configuration (Project) → Consumption (Customer) → Execution (Worker Agent). It implies each layer should have its own table/service boundary and that "intelligence" (reasoning, decisions, learning) must not live inside a business-logic module.
- **Gap vs. current repo:** Partially present but not cleanly layered. The 25-file `src/lib/engines/` deterministic computation library is functionality/capability-layer code (GST, payroll, accounting math) correctly kept out of any single "module" — a good sign. But there is no distinct "Enterprise Cognitive Graph" (see Study 3 below — no knowledge-graph implementation was found anywhere in `src/`), so the "stores enterprise knowledge" role in this ownership split currently has no home. Worker agents (`worker_agents` table) do exist as an execution layer distinct from application code, which matches this principle well.
- **Implementation recommendation:** Treat "Enterprise Cognitive Graph" as the single biggest missing piece implied by this section — flagged again under principle 3 and Study 3's Knowledge Objects concept. Any future implementation plan should scope this before adding more capability/module code, since this section states intelligence-layer gaps compound if capability keeps growing without a knowledge substrate.

What exists instead: Security/Governance intelligence is real and centralized (`policy-enforcement-engine.ts`, `SENTINEL.yaml` — confirmed by direct read of `VERIDIAN_AI_CONSTITUTION.md` §1–13 which cites exact enforcement file:line references). Conversation, Decision, Planning, Learning, Reasoning, Prediction, Innovation, Memory intelligence have no dedicated centralized service — each is either absent or implemented ad hoc per-feature (e.g., LLM calls scattered through `chat-service.ts` and individual API routes rather than routed through one Decision/Reasoning service).

Since 8 of 13 listed intelligences have no current home, the realistic near-term move is not "build all 13 services" but to consolidate LLM-calling code (currently spread across `llm-client.ts` callers) behind fewer, more clearly-named service boundaries as new features touch it — an opportunistic refactor discipline rather than a big-bang build.

### 5. Universal Capability Model / Capability Library (source lines 159–187)
- **Understanding:** Every capability (Authentication, OCR, Approval Workflow, GST Calculation, Digital Signatures, etc.) should become a reusable, registered platform asset in a "VERIDIAN Capability Library" — never recreated once it exists.
- **Architecture/Schema implications:** Requires a capability registry (table or service) queryable before writing new code, ideally with semantic search so near-duplicate capabilities surface instead of being silently re-implemented.
- **Gap vs. current repo:** **This is the strongest concrete match found in the entire study range.** `src/lib/services/capability-registry-service.ts` is real and functioning: it indexes 4 entity types (`worker_agent`, `automation_rule`, `module`, `prompt_pattern`) as embeddings, exposes `findSimilarCapabilities()` and `auditDuplicateCapabilities()` (embedding-similarity duplicate detection at threshold 0.92), with a UI at `src/app/(app)/capability-registry/page.tsx` and API routes at `src/app/api/capability-registry/{duplicates,backfill}`. It's a thin layer over the generic `embeddings` table (`schema.ts:523`), not a dedicated table with the richer metadata the doc later demands (owner, dependencies, usage metrics, quality score, compatibility — see principle 16), but the core "search before you build, detect duplicates" mechanic this section calls for is genuinely built and live.
- **Implementation recommendation:** Extend rather than rebuild: (1) widen `CAPABILITY_ENTITY_TYPES` beyond the current 4 (worker_agent, automation_rule, module, prompt_pattern) to cover more of the doc's example list (Authentication, OCR, Digital Signatures, etc. as first-class entries), (2) add the metadata fields principle 16 requires (owner, version, dependencies, quality score) to whatever the registry indexes, (3) wire `auditDuplicateCapabilities()` into the AI Coding Directive workflow (principle 14) so any AI agent proposing new code is required to call it first.

### 6. Universal Module Library (source lines 189–211)
- **Understanding:** Lists 17 example business modules (Accounting, HRMS, CRM, Construction, Inventory, Payroll, Compliance, Audit, etc.) that should each expose reusable services which projects assemble rather than rewrite.
- **Architecture/Schema implications:** Implies each module should be a service with a stable API boundary, independently versionable, consumed by multiple products.
- **Gap vs. current repo:** Partial. There's no generic "module-as-a-service" abstraction layer. What exists: 25 domain-specific calculation files in `src/lib/engines/` (accounting-engine.ts, hr-engine.ts, payroll-engine.ts, crm-engine.ts, inventory-engine.ts, procurement-engine.ts, etc.) — these are computation libraries, not full reusable services with their own UI/API/permission surface. Separately, `module-rules-resolver.ts` provides scoped (org/client/project/user) business-rule overrides per module, which is a real but narrower mechanism (rule resolution, not full module encapsulation). The clearest evidence of module reuse across products is PROJEXA's 42-route `/api/v1/projexa/*` aliasing surface, which does demonstrate one product (PROJEXA) consuming VERIDIAN's underlying modules rather than rebuilding them — a real, working instance of this principle, even though the generalized "module library" abstraction itself doesn't exist as a distinct architectural layer.
- **Implementation recommendation:** Don't build a new module abstraction from scratch — instead formalize what PROJEXA's aliasing pattern already proves works (one canonical module, thin per-product API aliasing) as the documented pattern for future products, and audit whether the 25 `engines/` files could be promoted to full "modules" (adding API/permission/UI layers) where a second product needs them.

### 7 & 8. Universal Product Factory / Universal Project Factory (source lines 213–243)
- **Understanding:** Two paired principles: creating a new *product* should mean selecting existing modules/workflows/agents/integrations/dashboards/reports/permissions and applying branding/rules/subscriptions, with minimal new code (Product Factory). Creating a new *project* (customer implementation) should similarly be an assembly of configuration, module selection, workflow selection, agent selection, business rules, UI config, branding, permissions, integrations, reports, dashboards, and knowledge packs — "assembled, not programmed" (Project Factory).
- **Architecture/Schema implications:** Requires a metadata-driven configuration system: a "product definition" and "project definition" record that references (not copies) underlying modules/agents/workflows, plus a UI for assembling these without code changes.
- **Gap vs. current repo:** PROJEXA is direct, real evidence this pattern is being followed at the product level (it was built by aliasing existing VERIDIAN modules via `/api/v1/projexa/*` rather than a rewrite — matching the "Product Factory" idea closely, per prior session notes and confirmed by the 42-route aliasing surface found in `src/app/api/v1/projexa/`). At the project (per-customer) level, `module-rules-resolver.ts`'s org/client/project/user-scoped rule overrides are a real, if partial, instance of "assembled not programmed" configuration. No single "Project Factory" UI/wizard was found that lets someone assemble a full customer implementation (modules + workflows + agents + branding + permissions in one flow) — this remains conceptual.
- **Implementation recommendation:** Document PROJEXA's build as the reference case study for future "Product Factory" instances (a template/checklist derived from how it was actually done). A "Project Factory" configuration UI is a real, currently-missing piece of scope — likely a Wave-sized project once more modules exist to assemble.

### 14 & 17. AI Coding Directive and Platform Evolution Principle (source lines 325–336, 379–390)
- **Understanding:** Two paired process rules. The AI Coding Directive (14): before writing new code, developers/AI agents must check — does this capability exist? Can an existing module/agent/workflow/integration do it? Can metadata or business rules solve it? Only if every answer is "no" should new platform code be written. The Platform Evolution Principle (17): whenever a new customer requirement is implemented, first ask "can this become reusable?" — if yes, generalize it, document it, add it to the Capability Library, expose it via API, make it available platform-wide, so the platform continuously gets richer. Together they form a "search-first, generalize-after" discipline bracketing every piece of new work.
- **Architecture/Schema implications:** Both are process gates, not data models — but both presuppose the Capability Registry (principle 5) is queryable and actually consulted as a required step, e.g., in CI or in agent tooling.
- **Gap vs. current repo:** The prerequisite (`capability-registry-service.ts` with `findSimilarCapabilities()`/`auditDuplicateCapabilities()`) exists, as established under principle 5. What's not confirmed for either directive is that the check is *mandatory* anywhere — no CI step, pre-commit hook, or agent-tooling gate was found that forces a capability-registry lookup before new code is written, and no confirmed process step requires generalization-and-registration before a customer-specific feature ships. `AGENTS.md`'s Operating Rules govern PR/branch process but mention neither.
- **Implementation recommendation:** Low-cost, high-value, and the two directives should be implemented as one change: add a step to the AI agent operating rules (`AGENTS.md`) and/or a CI/PR-template checklist item requiring (a) a `findSimilarCapabilities()`/`auditDuplicateCapabilities()` call be shown before new service files are added, and (b) a "could this be generalized and registered?" checkbox before a customer-specific feature merges. This directly closes the gap between "the tool exists" and "the tool is used" for both principles at once.

### 15. Universal Dependency Rules (source lines 338–347)
- **Understanding:** No module may directly depend on another; every interaction must go through Platform Services, Capability APIs, Worker Agents, Event Bus, the Cognitive Graph, or the Workflow Engine — for loose coupling and scalability.
- **Architecture/Schema implications:** Requires either enforced architectural boundaries (module federation, dependency-cruiser/lint rules) or, at minimum, an actual Event Bus for cross-module signaling.
- **Gap vs. current repo:** **Not enforced.** No architectural lint config (e.g., dependency-cruiser, ESLint boundary rules) was found restricting cross-module imports. No Event Bus exists (confirmed under principle 3). Since Next.js/Drizzle apps commonly import across "modules" freely by file path, there is currently no structural barrier preventing exactly the direct coupling this principle forbids — it can only be a code-review convention today, not a system guarantee.
- **Implementation recommendation:** Two independent, additive fixes: (1) introduce a lightweight architectural-boundary lint rule (even a simple ESLint `no-restricted-imports` per top-level module directory) as a first, cheap enforcement step; (2) treat "build the Event Bus" as its own scoped project, since several other principles (3, 15) depend on it and none currently reference one.

### 16. Universal Asset Registry (source lines 349–378)

### 19 & 20. Engineering Principle and Foundational Architecture Principle + Capability Engineering addendum (source lines 406–421)
- **Understanding:** Principle 19 states the platform-wide engineering mantra: "Build once. Configure many. Reuse everywhere. Govern centrally. Improve continuously." — no capability should be built exclusively for one project if it can reasonably become a reusable platform asset. Principle 20 is a consolidating paragraph restating that VERIDIAN AI OS is the permanent cognitive core and everything else a plug-in, intelligence never fragmented. As a named discipline, "Capability Engineering" (20) doesn't appear anywhere in the repo, but its mechanical prerequisite (the Capability Registry, principle 5) is real — the tooling to do the right thing often exists; the discipline/process requiring its use does not.

### Study 1, Level 1 — Mode Pills and Option Selector (source lines 422–423)
- **Understanding:** Proposes that ~90% of user intent clarity can come from offering clickable "mode pills"/option selectors rather than free text, before any prompt-compression technique is even needed.
- **Architecture/Schema implications:** A UI-layer pattern (button/pill selector components feeding structured params) that should precede/replace free-text input wherever a request is common and enumerable.
- **Gap vs. current repo:** Not confirmed as a systematic UI pattern. The repo has standard forms and buttons throughout its UI (shadcn/ui-based), but no evidence of a deliberate "mode pill" pattern specifically designed to short-circuit free-text-to-LLM flows was found (this is consistent with the absence of an Intent Engine — see Study 1 Level 5 below).

### Study 1, Level 2 — Safe-Word Stripping and Phrase Normalization (source lines 424–500)
- **Understanding:** Lists categories of conversational filler (greetings, politeness, hedges like "I think"/"maybe", ways of addressing the AI, meta-phrases like "I want you to") that can usually be stripped or normalized without losing intent, with worked examples ("I want you to please explain to me how GST filing works" → "Explain GST filing").
- **Architecture/Schema implications:** A pure text-preprocessing function/service — stateless, deterministic, sits in front of any LLM call.
- **Gap vs. current repo:** **Not found.** No prompt-normalization or filler-stripping utility exists anywhere in `src/lib`. Chat/LLM-calling code (`chat-service.ts`, `llm-client.ts`) sends user text through without this kind of preprocessing step, as far as could be confirmed.
- **Implementation recommendation:** This is one of the cheapest, most self-contained wins in the whole document — a single pure function (regex/word-list based) inserted before any LLM call. Low risk given it's explicitly scoped to filler removal, not semantic rewriting. Should be built alongside Study 1 Level 6's "words you should NOT remove" list (negations, quantifiers) as a paired safety constraint.

### Study 1, Level 3 — Domain-Specific Aliasing (source lines 501–527)
- **Understanding:** Proposes a VERIDIAN-specific abbreviation dictionary (Construction ERP→CERP, Project Dashboard→PD, Minutes of Meeting→MOM, Purchase Order→PO, BOQ, VO, etc.) to compress recurring domain terms in prompts, expanded internally only when needed — claimed to save significant tokens at scale.
- **Architecture/Schema implications:** A lookup-table service (alias↔full-term), likely versioned per domain/industry pack given later Study sections mention industry-specific libraries (Construction vs. Accounting).
- **Gap vs. current repo:** Not found as a token-optimization mechanism. Abbreviations like BOQ, RFI, PO are used naturally throughout the PROJEXA construction-domain code (e.g., BOQ appears extensively in `/api/v1/projexa/*` routes) but there's no deliberate compression *dictionary* or alias-expansion service — these are just the domain's natural vocabulary, not an engineered token-savings layer.
- **Implementation recommendation:** Lower priority than Level 2's filler-stripping (smaller marginal token savings per the source's own claim ordering), and should be deferred until there's a real LLM-cost-per-request metric to justify it (none was found — see Study 2 gap below on token accounting).

### Study 1, Level 4 — Session Memory / Incremental Context (source lines 529–538)
- **Understanding:** Instead of resending full context (company info, project details) on every prompt, send a durable `Workspace Context ID` once, then only deltas — described as the "biggest savings" lever.
- **Architecture/Schema implications:** Requires server-side session/context state keyed by a short ID, with the LLM-calling layer resolving the ID to full context only when needed rather than the client (or app code) re-serializing everything per call.
- **Gap vs. current repo:** Partial infrastructure exists but not this specific pattern. `llm-response-cache.ts` caches full responses keyed by a hash of org+provider+model+prompt+messages (exact-match, 24h TTL) — a related but different mechanism (caches outputs, not incremental context). No evidence of a context-ID system that lets the app send `WorkspaceContextID: WS-1284` instead of re-sending full workspace/company state on every LLM call.
- **Implementation recommendation:** This is likely the single highest-ROI item in the entire Study 1–2 range per the source's own claim, and is currently unbuilt. Recommend scoping as: a server-side context store keyed by workspace/session ID, with LLM-call-building code resolving it just before the call (not before) so the token savings are real, not just moved. Depends on nothing else in this range — good candidate for early implementation.

### Study 1, Level 5 — Intent Engine (source lines 540–556)
- **Understanding:** Instead of sending raw text to the LLM, convert user input into structured intent objects (e.g., `{"intent": "create_purchase_order", "vendor": "ABC", "item": "cement", "qty": 50}`) — shorter, clearer, more reliable than free text.
- **Architecture/Schema implications:** Requires an NLU/classification layer upstream of any LLM call, plus a defined intent schema per supported action (likely one JSON schema per intent type, versioned).
- **Gap vs. current repo:** **Confirmed not found.** A repo-wide search for "intent"/"Intent" in `src/lib` returns zero conceptual matches (the few hits are unrelated: schema comments, an unrelated `classification.ts` for compliance-item categorization, not user-intent parsing). This is one of the most concretely absent pieces of infrastructure in the entire studied range, and it's a named prerequisite for several later Study 4/5 principles (Slot Filling, Intent Routing, Intent Confidence).
- **Implementation recommendation:** High-value, but sequence-dependent — recommend building this only after Session Memory (Level 4) and the deterministic Rule Engine (Level 6, below) are in place, since the Intent Engine's main value (routing to deterministic code instead of an LLM) is worthless without those deterministic targets to route *to*. This is elaborated much further in Study 4/5's Conversation Engine material below — treat as one initiative, not built twice.

### Study 1, Level 6 — Deterministic Software Before LLM (source lines 558–599)
- **Understanding:** Many requests (calculators, GST/TDS calculations, date arithmetic, currency conversion, SQL queries, sorting/filtering/totals) don't need an LLM at all — use deterministic code and only invoke the LLM for genuine natural-language understanding/generation. Also lists words that must never be stripped by any compression step (not, never, don't, except, unless, only, without, before, after, if, else, should, must, may, all, any, every, first, last) since removing them changes meaning.
- **Architecture/Schema implications:** Confirms the Intent Engine (Level 5) must route to a deterministic compute layer as its primary target, with the LLM as fallback only. The "words not to remove" list is a hard constraint on the Level 2 filler-stripper.
- **Gap vs. current repo:** **The deterministic-compute half is strongly built.** This is exactly what `src/lib/engines/` (25 files: `gst-engine.ts`, `tds-engine.ts`, `payroll-engine.ts`, `accounting-engine.ts`, `mathematical-engine.ts`, etc., ~1,914 lines total, using `decimal.js` for precision) already provides, plus the `computation_engines` table (`schema.ts:~6940`) which registers engines by `engineKey`/`category`/`status` ('implemented'/'partial'/'not_started') — matching this section's spirit closely. **However, it is not wired into anything resembling the Level 5 Intent Engine or a "Need LLM?" routing decision** — these engines are called directly by specific feature services (e.g., `gst-reconciliation-service.ts`), not by a general request-router that decides LLM-vs-deterministic per incoming user request. The "words not to remove" safety list has no corresponding implementation to protect, since Level 2's stripper doesn't exist yet either.
- **Implementation recommendation:** The building block (engines) is done; the missing piece is the router that decides, per incoming request, whether to call an engine or an LLM — this is the same gap as Level 5's Intent Engine and should not be built as a separate effort.

### Study 1 — Recommended Token Utilization Engine (TUE) Pipeline (source lines 601–615)
- **Understanding:** Proposes bundling all of Study 1's levels into one named subsystem — Mode Pills → Intent Detection → Rule Engine → Prompt Normalizer → Context Manager → Domain Dictionary → Semantic Compression → Prompt Cache → Model Router — arguing this combined pipeline saves far more tokens than simple stop-word removal alone.
- **Architecture/Schema implications:** A single orchestration pipeline/middleware sitting in front of every LLM call, composed of the individually-gapped pieces above.
- **Gap vs. current repo:** No such orchestrator exists. Individual pieces are unevenly built: Rule Engine (deterministic engines) exists; Prompt Cache exists narrowly (`llm-response-cache.ts`, exact-match only, not semantic); Model Router exists narrowly (`llm-client.ts`'s `LLMFallback` type supports primary/secondary provider fallback, but this is failover, not cost/complexity-based routing); Mode Pills, Intent Detection, Prompt Normalizer, Context Manager, Domain Dictionary, and Semantic Compression are all absent.
- **Implementation recommendation:** This TUE pipeline is the correct target shape for consolidating Study 1's individually-gapped items — recommend it as one epic with the sub-items above as its tickets, built roughly in the dependency order already implied by the levels (Prompt Normalizer and Context Manager first since they're standalone and cheap; Intent Engine once there's something deterministic to route to; Model Router and Semantic Compression last since they need real usage data to tune).

### Study 3 — Learning as Software, Not Prompts (source lines 726–778)
- **Understanding:** Argues the AI OS shouldn't "learn" by accumulating more text in prompts (which scales token cost linearly with knowledge) but by converting repeated reasoning into durable software artifacts — new rules, agents, processes, workflows, SQL queries, APIs, modules, decision trees — executed with zero LLM tokens thereafter. Diagrams a "VERIDIAN Cognitive AI OS" flow: User → Intent Engine → Knowledge Graph → Decision Engine → Worker Agents → Business Rules → LLM (only if reasoning required), contrasted against a naive "traditional AI" flow where everything goes through the LLM with ever-growing prompts.
- **Architecture/Schema implications:** Requires the Knowledge Graph (again — see principles 2/3/12 above) as the central artifact store, plus a Decision Engine positioned between the Knowledge Graph and Worker Agents/LLM.
- **Gap vs. current repo:** The philosophy is partially realized asymmetrically: the "convert learning into a software artifact" half is real for one artifact type — `worker_agent_learnings` is a genuine table capturing agent-level learnings as structured data rather than re-prompted context. But there's no Decision Engine or Knowledge Graph sitting between Intent and Worker Agents (both absent, as established repeatedly above), so the full pipeline this section describes doesn't exist end-to-end — only its terminal node (worker agents themselves) is real.
- **Implementation recommendation:** This section is the clearest statement yet of why the Knowledge Graph gap (flagged three times now — principles 2, 3, 12) is the single highest-leverage missing piece in this whole document range: nearly every subsequent Study 3–5 idea (Agent Factory, Process Factory, Module Factory, Knowledge Objects, Hierarchical Memory) presupposes it exists. Recommend it becomes the first major implementation target once this 6-part study concludes.

### Study 3 — Agent Factory, Process Factory, Module Factory (source lines 819–895)
- **Understanding:** Three parallel "factory" concepts: Agent Factory generates a persistent worker agent (with tests, docs) once for a new domain rather than re-reasoning as that expert every time; Process Factory recognizes recurring multi-step user actions (e.g., Approve Invoice → Send Email → Update ERP → Notify Finance) and converts them into a reusable, LLM-free "Run Process #483"; Module Factory infers, from repeated requests for something like "Equipment Maintenance," the tables/UI/APIs/permissions/reports/agents needed and proposes a whole new module.

Process Factory and Module Factory have no landing targets at all (no workflow-definition table, no module registry — both confirmed absent above).
- **Implementation recommendation:** Sequence-dependent on the still-missing Workflow Engine and Module Registry — recommend scoping Process Factory only after a basic Workflow Engine exists (itself implied by principle 15's Dependency Rules), and Module Factory only after principle 16's Asset Registry is generalized beyond worker agents.

### Study 3 — Hierarchical Memory and Token Budget Manager (source lines 973–1019)
- **Understanding:** Proposes layered memory (Session → Workspace → Domain → Agent → Enterprise Knowledge → Global Knowledge, each queried only when relevant) and a "Token Budget Manager" that treats tokens like CPU/RAM — for every request, classify intent, decide if an LLM is needed, pick small/medium/frontier model tier, apply a max token budget, cache the result, and learn.

### Study 4 — Conversation Intelligence Engine (CIE) Architecture (source lines 1093–1176)
- **Understanding:** Proposes a subsystem where the LLM doesn't answer directly; instead User → Intent Engine → Conversation Engine → Decision Engine → Workflow Engine → Task Engine → Worker Agents, with the Conversation Engine internally asking "what information is missing / what should I ask next / what options should I show / what can be automated" before invoking a worker agent. VERI should constantly ask itself: can I reduce typing, thinking, mistakes; can I automate this.
- **Architecture/Schema implications:** A 6-stage request pipeline (Intent → Conversation → Decision → Workflow → Task → Worker Agent) — the most complete architectural diagram for the "conversation layer" specifically in this range.
- **Gap vs. current repo:** None of the 6 named engines exist as distinct services. - **Implementation recommendation:** This diagram is the single best target-architecture reference for the whole "Conversation Engine" gap that recurs across Study 4 and 5 — recommend it (or Study 5's near-identical Conversation Execution Engine, below) be the canonical spec once implementation begins, rather than treating each Study section's slightly different phrasing as a separate system.

### Study 4 — Suggestion, Approval, Permission, and Clarification Engines (source lines 1177–1261)
- **Understanding:** Four related sub-engines: Suggestion Engine proactively observes patterns and offers automation ("85% of your projects have a kickoff meeting — should I auto-schedule?"); Approval Engine never silently makes business decisions, instead presents recommended actions with estimated impact and Yes/No/Details buttons; Permission Engine, instead of a bare "Access Denied," offers to request approval from the right person; Clarification Engine asks a multiple-choice question ("Which report?") instead of leaving the request ambiguous.
- **Architecture/Schema implications:** Each implies a distinct service: Suggestion Engine needs usage-pattern analytics (ties to Study 3's "Learning Through Metrics" gap); Approval Engine needs a structured recommendation-with-impact-estimate object type; Permission Engine needs to hook into the existing RBAC/RLS layer to offer an approval-request action on denial rather than just failing.
- **Gap vs. current repo:** Permission Engine's *foundation* is strong — RBAC/RLS is genuinely enforced (`requireAuth()`, `hasRole()`, RLS per the Constitution) — but the "offer to request approval" UX layer on top of a denial doesn't exist; denials are just denials today. Approval Engine: no structured recommendation-with-impact object type was found. Suggestion Engine: no pattern-detection-to-suggestion service exists (same gap as Study 3's Learning Through Metrics). Clarification Engine: same gap as the Intent Engine (Study 1 Level 5) — no missing-slot detection exists to know what to clarify.
- **Implementation recommendation:** Permission Engine is the cheapest win here — since RBAC/RLS already correctly identifies "you don't have permission," the only new work is adding an "offer to request approval from role X" action on top of an existing, already-correct denial path. The other three depend on Intent Engine and Knowledge Graph work flagged above.

### Study 4 — Progressive Disclosure, Prediction, Memory, Confidence, Explainability Engines (source lines 1262–1332)
- **Understanding:** Progressive Disclosure: ask one question at a time, not ten at once. Prediction Engine: after "Project Created," proactively suggest the next likely steps (Invite Members, Upload Drawings, etc.) based on what most teams do next. Memory Engine: remember user preferences (e.g., last report format was "PDF, Landscape, Detailed") and offer to reuse them. Confidence Engine: never claim false certainty — state confidence explicitly (e.g., "98% confident this vendor is a duplicate"). Explainability Engine: every recommendation should state its reason, impact, and expected outcome in structured form.
- **Architecture/Schema implications:** Prediction Engine needs a "next likely action" model (could start as a simple co-occurrence/frequency table per intent, not necessarily ML). Memory Engine needs a user-preference store. Confidence Engine needs every automated decision/recommendation to carry a confidence score field. Explainability needs a structured recommendation object (reason/impact/risk/savings fields), same shape as Study 5 Principle 10's "Explainability Templates."
- **Gap vs. current repo:** No user-preference store was found (no "last report format" style persisted setting). No confidence-score field convention was found on any recommendation-producing code path. `accuracyScore` on `worker_agents` is the closest existing analog to a "confidence" concept, but it's agent-level (a historical accuracy metric), not per-recommendation. No "next likely action" prediction service exists.
- **Implementation recommendation:** These five sub-engines are lower priority than the Intent/Conversation Engine core, since they're refinements layered on top of a working guided-conversation system that doesn't exist yet — sequence after the core Conversation Engine work, not before.

### Study 4 — Conversation State Machine and Personality Layer (source lines 1333–1411)
- **Understanding:** Every conversation should move through explicit states (Greeting → Identify Intent → Collect Missing Information → Validate → Generate Plan → Ask Approval → Execute → Monitor → Summarize → Suggest Next Action) and never skip states. VERI should have one consistent enforced personality (professional, friendly, concise, proactive, transparent, always confirms before impactful actions) — enforced by the Conversation Engine itself, not re-described in every prompt. The section closes by naming a "Conversational Intelligence Layer (CIL)" composed of Intent Engine, Conversation Planner, Context Engine, Suggestion Engine, Clarification Engine, Approval Engine, Permission Engine, Workflow Navigator, Explanation Engine, Personality Engine, and Next Best Action Engine.
- **Architecture/Schema implications:** A finite state machine per active conversation (state stored server-side, not inferred from chat history each time) plus a Personality Engine that post-processes/constrains all VERI-authored output centrally (rather than via per-prompt system-prompt text).
- **Gap vs. current repo:** No conversation state machine was found (no `stateMachine`/`conversation_state` table or equivalent). Personality/tone consistency today, if enforced at all, would be via system-prompt text repeated per call site rather than a centralized Personality Engine — consistent with the general finding that LLM calls are made directly by feature code rather than through one governed conversational layer.
- **Implementation recommendation:** This CIL list is essentially a renamed, slightly reorganized version of Study 4's earlier CIE diagram and Study 5's CEE below — all three should be treated as one target architecture (pick one canonical name and engine list) rather than three separate specs, to avoid building overlapping subsystems under different names.

### Study 5 — LLM as Last Resort: The "Two Brains" Model (source lines 1413–1436)
- **Understanding:** Reframes the goal from "optimize LLM cost" to "optimize LLM dependency" — the user should experience natural conversation, but internally 90–98% of interactions should be deterministic software ("Software Brain"), with the "LLM Brain" invoked only ~5% of the time, invisibly to the user.
- **Architecture/Schema implications:** This is the thesis statement Study 1–4's individual mechanisms (Intent Engine, Rule Engine, deterministic `engines/`, Conversation Templates) all serve — no new implication beyond unifying them under one measurable target ratio (LLM-vs-software call ratio).
- **Gap vs. current repo:** No metric currently exists to even measure this ratio (ties to the Study 2 gap on token accounting/tiktoken-style instrumentation) — so beyond the qualitative gaps already itemized, there's a measurement gap too: VERIDIAN cannot today state what fraction of its interactions are LLM vs. deterministic.
- **Implementation recommendation:** Recommend this ratio become an explicit, tracked metric once basic token accounting exists (Study 2 gap) — it's a good north-star KPI for evaluating whether the eventual Intent/Conversation Engine work is succeeding.

### Study 5, Principles 1–8 — Zero-Token Conversation Mechanics (source lines 1437–1585)
- **Understanding:** Eight concrete techniques for eliminating LLM calls on common turns: (1) don't ask an LLM to say things already decided in code (e.g., render "Let's create a new project" from a template, not a generated reply); (2) Conversation Templates — every ERP process has finite states, each with predefined text; (3) Conversation Components — generate UI (file upload widgets), not text; (4) Business Conversation Library — thousands of reusable fixed messages ("Invoice Approved", etc.); (5) Intent Routing — known intents go straight to software, only unknowns reach the LLM; (6) Slot Filling — ask only for the fields actually missing; (7) UI instead of words — offer buttons (Sales/Finance/HR/GST/Construction) instead of asking "which report?"; (8) Predictive UI — pre-show buttons for what this business type usually does next.
- **Architecture/Schema implications:** All eight point to the same missing artifact: a message/template library with IDs (foreshadowing Study 5's later "MSG-00431" concept) plus the Intent Engine's slot-filling logic.
- **Gap vs. current repo:** **Confirmed absent** — direct search for `MSG-`, `messageTemplate`, `conversationTemplate` across `src/` returned zero results. None of these 8 techniques have a supporting implementation today; all conversation output currently appears to be LLM-generated per call rather than template-selected.
- **Implementation recommendation:** This is the most concrete, buildable spec for the Conversation Template Library in the entire document range — recommend it as a discrete, sequenceable deliverable: (a) define a `conversation_templates` table (id, category, text, variables, industry pack), (b) seed it with the "Layer 1 Micro Templates" from later in Study 5 (see below) as the first content, (c) wire Intent Routing (principle 5) to check this library before ever calling an LLM.

### Study 5, Principles 9–16 — Grammar Engine, Explainability/Decision Cards, Structured Outputs (source lines 1609–1793)
- **Understanding:** Further zero-token patterns: Grammar Engine (fill a template like "{document} approved." rather than generating the sentence); Explainability Templates (structured Recommendation/Reason/Impact/Risk/Savings/Approve fields, not prose); Decision Cards (structured approve/cancel/details UI instead of a conversational question); a rule that AI is invoked only for genuine ambiguity/complex reasoning/summarization/writing/negotiation/unknown workflows; Intent Confidence (if >98% confident, never call the LLM); a Conversation State Machine storing state server-side (not relying on LLM memory); Tiny Prompts (send `WorkspaceID=51`, not full context — same as Study 1 Level 4); Structured Outputs (ask the LLM for JSON like `{next_question, reason, confidence, actions}`, not prose, then render it in VERI's style). Introduces the "Conversational Virtual Machine (CVM)" concept: the LLM produces conversation *instructions* (structured intent/missing-fields/next-step JSON), never final UI/wording — software renders everything. Closes with a proposed "Conversation Execution Engine (CEE — note: same acronym Study 3 used for Cognitive Evolution Engineering, a naming collision in the source)" comprising Intent Engine, Workflow Engine, Conversation Planner, Template Library, UI Composer, Decision Engine, LLM Gateway, Response Renderer, Learning Engine.
- **Architecture/Schema implications:** The CVM concept is architecturally important and specific: it draws a hard line that the LLM's only allowed output shape is structured JSON (intent/missing-fields/next-step), never rendered text/UI — rendering is always a deterministic software step. This is a stronger, more implementable version of the general "Conversation Engine" idea than earlier sections' looser diagrams.
- **Gap vs. current repo:** Not implemented — current LLM calls (per `llm-client.ts` usage patterns found) appear to return text/content directly rather than being constrained to a structured intent-JSON contract with a separate rendering step. **Note the source document itself reuses the abbreviation "CEE" for two different concepts** (Study 3's Cognitive Evolution Engineering and Study 5's Conversation Execution Engine) — worth flagging explicitly so a future consolidated implementation plan picks distinct names and doesn't conflate them.
- **Implementation recommendation:** The CVM's "LLM only ever returns structured JSON, software always renders" rule should be adopted as a hard engineering constraint the moment any Conversation Engine work begins — it's cheap to enforce from day one (a response schema + validation step) and expensive to retrofit later once callers assume free-text LLM responses.

### Study 5 — Conversation Library Layers 1–4 and the Conversation Knowledge Base (source lines 1795–2082)
- **Understanding:** Elaborates the message-library idea into four concrete layers: Layer 1, Micro Templates (thousands of tiny reusable sentences: "Done.", "Saved successfully.", "Invoice approved."); Layer 2, Conversation Blocks (reusable patterns: Confirmation, Information Collection, Approval Request, Warning, Error, Summary, Next Steps); Layer 3, Workflow Conversations (a whole workflow's dialogue sequence pre-defined, e.g. Create Project → Ask Client → Ask Budget → Validate → Save → Offer Next Steps); Layer 4, Industry Libraries (Construction's dialogue sequence differs from Accounting's). Proposes versioning conversations (v1/v2/v3) and measuring click-rate/satisfaction/completion-time to iteratively improve them; Adaptive Conversation (same workflow, different wording for beginner vs. expert users); and finally a full "Conversation Knowledge Base (CKB)" with 50,000+ entries across Micro Responses, Conversation Blocks, Workflow Dialogues, Industry Templates, and various message-type categories, each with an ID (e.g., `MSG-00431`). Closes by proposing thinking of this as a "Conversation OS" with sub-components: Conversation Grammar, Conversation Components, Conversation Templates, Workflow Dialogues, Personality Engine, UI Components, Suggestion Engine, Approval Engine, Clarification Engine, Adaptive Conversation Engine, Industry Packs, Localization Engine — with the LLM's role finally reduced to ambiguity/creative-generation/complex-reasoning only, and >90% of interactions expected to be served by this Conversation OS.
- **Architecture/Schema implications:** This is the fullest, most schema-ready specification in the whole 2084-line range: a `conversation_templates`/`messages` table with `id` (MSG-xxxxx), category/layer (micro/block/workflow/industry), version, variables, industry-pack tag, and A/B-style usage metrics (click rate, satisfaction, completion time) — directly reusable as the schema for the Study 5 Principles 1–8 recommendation above. The "Adaptive Conversation" (beginner/expert mode) implies a per-user or per-role skill-level field driving which template variant renders.
- **Gap vs. current repo:** Confirmed absent in every respect (same search as above: zero `MSG-`-style IDs, no conversation template tables, no industry-pack-scoped dialogue library). This is the single largest concrete, well-specified, and entirely unbuilt subsystem identified across the full Part 1 study range.
- **Implementation recommendation:** Recommend this become the flagship deliverable once this 6-part study concludes and a consolidated plan is written — it's unusually well-specified for a document at this level of abstraction (explicit ID scheme, four clear layers, explicit metrics to track). Suggested build order: (1) `conversation_templates` schema + Layer 1 Micro Templates seed data (cheapest, highest immediate token savings per Study 5's own "two brains" 90-98% claim), (2) Layer 2 Conversation Blocks, (3) wire Intent Routing to check this library before any LLM call (depends on the still-missing Intent Engine from Study 1 Level 5 — build that dependency first or in parallel), (4) Layer 3/4 workflow- and industry-specific dialogues once PROJEXA and other verticals have enough usage data to know their real workflows, (5) versioning/metrics/adaptive-mode as a later refinement once Layer 1–3 are live and generating usage signal.

### 3. CSV 201 — Conversation Operating System: Vision & Philosophy (source lines 2170–2235)

- **Understanding:** COS is defined as a deterministic software platform (not an LLM wrapper) that manages conversations, reduces user effort/typing/ambiguity/tokens, and increases automation — with the LLM as "only one service inside COS," never the OS itself. Core philosophy: "Human First" (VERI acts like a colleague, not a chatbot), "Software First / AI Last" (only invoke AI for reasoning/creativity/summarization/language understanding), "Guided Conversation" (never let the user wonder what to type), "Click Before Typing" (buttons/menus/cards/forms/autocomplete over free text), "Human In Control" (never silently make critical decisions), and "Transparency" (every recommendation explains why/impact/confidence/risk).
- **Architecture/Schema implications:** Implies a hard architectural separation: a deterministic orchestration layer that owns UI/state/decisions, with LLM calls as a bounded, optional leaf dependency — the opposite of "send everything to the model and let it decide."
- **Gap vs. current repo:** Partially inverted today. `src/lib/services/chat-service.ts`'s `sendMessage`/`generateAiReply` flow calls the LLM (`callLLM` from `src/lib/llm-client.ts`) fairly directly for conversational replies once a message is sent — there is no intent-first "try software, only escalate to LLM if needed" gate in the chat path itself. The closest existing "software first" pattern is the 247-engine deterministic computation registry in `src/lib/engines/` (EMI, depreciation, gratuity, GST-split, PAN/GSTIN/IFSC validators) — but that's invoked as computation tools, not as a pre-LLM decision layer inside the chat/conversation flow. `policy-enforcement-engine.ts` gates *whether* an LLM call is allowed (domain/purpose), not *whether software could answer instead*.
- **Implementation recommendation:** Introduce a "Decision Engine" step ahead of `generateAiReply` in `chat-service.ts` that checks intent against known deterministic capabilities (engines registry, direct CRUD, workflow actions) before falling through to the LLM. This depends on subsection 8 (Intent Intelligence) existing first — without structured intents, there's nothing to route on.

### 4. CSV 201 — Core Architecture Pipeline & Golden Rule (source lines 2270–2403)

- **Understanding:** Defines a fixed pipeline: Human → Conversation UI → Conversation Planner → Intent Engine → Decision Engine → Workflow Engine → Software Engine → (Need AI? No→Execute / Yes→LLM Gateway → Worker Agent) → Response Renderer → User. The "Golden Rule": the LLM is never allowed to talk to the user directly — it must return structured data (e.g., `{Intent: CreateInvoice, MissingFields: [...], RecommendedNextAction: SearchVendor}`) which COS converts into UI. Also states the "Core Rule": never ask the user for information VERIDIAN already knows (e.g., infer company name rather than asking).
- **Architecture/Schema implications:** This is the single most consequential architectural claim in this section — it implies every LLM call in the platform should return structured JSON (intent objects, not prose), with a separate deterministic rendering layer turning that into UI. It also implies a "known facts" store queried before ever prompting the user (workspace/org profile lookups).
- **Gap vs. current repo:** Directly violated in the current chat implementation. `generateAiReply` in `chat-service.ts` calls `callLLM` (see `src/lib/llm-client.ts`) and the raw text response is stored as message `content` and shown to the user verbatim — this is prose-to-user, not structured-data-to-renderer. There is no "Response Renderer" translating LLM JSON into cards/buttons. `resolvePromptTemplate` (prompt-os-resolver.ts) governs prompt *content* but not response *shape/parsing*. The "Core Rule" (don't ask what's already known) is honored in spirit by `resolveModelConfig`/org-scoped context injection patterns elsewhere in the codebase (e.g., org name is looked up, not asked), but there's no general-purpose "known facts" resolution service.
- **Implementation recommendation:** Define a structured LLM response contract (e.g., `{intent, missingFields, recommendedActions, confidence}`) for conversational endpoints, and a renderer that maps it to existing UI primitives (shadcn `Card`, `Button`, forms already used elsewhere in `src/components/`). Large lift — depends on Intent Engine (subsection 8) and Components Library (subsections 9–13) both existing.

### 7. CSV 202 — Conversation Knowledge Base: Vision, Philosophy & Reuse-Before-Generate (source lines 2579–2660)

- **Understanding:** CKB is the "master repository of all reusable conversational assets" — the idea that every high-quality conversation VERI has ever produced becomes a reusable, versioned knowledge object, so future similar situations are assembled from existing components rather than regenerated. Philosophy: "Reuse Before Generate" — before any LLM call, search CKB, then Workflow Library, then Industry Pack, then Conversation Components; only then invoke AI. The architecture pipeline is Conversation Request → Intent → Workflow → Conversation Planner → CKB → Conversation Components → Template Engine → Variable Injection → Rendered Response, with the LLM only invoked if no suitable template exists.
- **Architecture/Schema implications:** Requires a template store with variable substitution (`{UserName}`, `{ProjectName}` etc.), a search/match layer keyed on intent+workflow+industry+role, and a template-vs-LLM confidence gate.
- **Gap vs. current repo:** `promptTemplates`/`promptVersions` (via `prompt-os-resolver.ts`) is a real, working partial implementation of exactly this idea — versioned, labeled (`production` label), keyed by `templateKey`, with an active-version lookup. This is a genuinely good match to CSV 202's "Versioning" (subsection 11 below) concept. However, it governs *LLM system prompts* only, not *user-facing conversational response templates* — there's no equivalent table for pre-built reply text/cards that bypass the LLM entirely. No "search CKB before calling LLM" gate exists in `chat-service.ts`; every AI thread reply goes straight to `callLLM`.
- **Implementation recommendation:** Extend the existing `promptTemplates`/`promptVersions` pattern (proven, in production) with a parallel `responseTemplates` concept for user-facing deterministic replies, plus a pre-LLM lookup step in `sendMessage`/`generateAiReply`. This is lower-risk than building CKB from scratch since the versioning/labeling infrastructure already exists and works.

### 8. CSV 202 — Knowledge Hierarchy: 6 Layers (source lines 2662–2745)

- **Gap vs. current repo:** No layered template system exists. `promptTemplates` is flat (keyed by `templateKey` + label only) — no industry/role/workspace override hierarchy. ### 10. CSV 202 — Component Assembly, IDs & Adaptive/Dynamic Conversation (source lines 2825–2912)

- **Understanding:** Messages are assembled from components in a fixed order (Greeting → Acknowledgement → Information → Question → Suggestion → Buttons → Footer, "like React components") rather than written as paragraphs. ### 28. CSV 206 — Conversation State Machine: Vision, Philosophy & Objectives (source lines 4065–4110)

- **Understanding:** CSM is "the execution engine responsible for controlling every conversation" — VERI must always know where a conversation started, its current state, previous decisions, pending info, completed actions, waiting approvals, and active workflows. Core philosophy: "Conversation is not text. Conversation is a state machine... Never rely on the LLM to remember context. The software owns the state." Objectives: eliminate repeated prompts, remove unnecessary memory tokens, resume conversations instantly, support long-running workflows, pause/resume, multiple simultaneous conversations, deterministic execution, and reduced LLM context requirements.
- **Architecture/Schema implications:** This is the strongest, most explicit statement in the whole section range against the "replay full chat history into every LLM call" pattern — directly contradicts what `buildConversationHistory` does today (see subsection 6).
- **Gap vs. current repo:** Confirmed gap, same root cause as subsection 6: `chat-service.ts` has no state machine — conversations are message lists with no `currentState`/`workflowId`/`pendingFields` tracking. The LLM effectively "remembers" context today because full history is resent, which is precisely the anti-pattern this section names.
- **Implementation recommendation:** This is the architectural fix that would resolve the token-cost issue flagged in subsection 6 properly (rather than just capping history length) — add explicit state tracking so context can be reconstructed from IDs instead of replayed text. Should be sequenced together with subsection 20 (Intent Engine) since state transitions are driven by recognized intents.

### 30. CSV 206 — Nested/Parallel State Machines & State Persistence (source lines 4268–4344)

- **Understanding:** Complex workflows (Project Creation → BOQ Upload → Drawing Upload → Task Assignment → Kickoff Meeting) are modeled as nested state machines, each a sub-machine tracked by the parent. VERI supports multiple simultaneous independent conversations (e.g., Project Creation + GST Filing + Payroll + Tender Review running in parallel for one user). State must persist across browser refresh, logout, power failure, server restart, and device changes — resume instantly. Users can interrupt a conversation mid-flow (e.g., ask "Show dashboard" during invoice creation) and it pauses safely, resuming later.
- **Architecture/Schema implications:** Nested state implies a parent/child relationship on conversation records (`parentConversationId`); persistence is naturally satisfied by any DB-backed state (already true since `conversations` is a Postgres table) — the gap is really about *whether* state is tracked at all (subsection 29), not about the persistence mechanism itself, which the DB already provides for free.
- **Gap vs. current repo:** Multiple simultaneous AI conversations: the current `ensureAiThread` logic in `chat-service.ts` finds or creates exactly **one** AI thread per user (`isAiThread: true`, looked up as a singleton) — this actively prevents the "parallel conversations" pattern described here; today there is one persistent AI thread per user, not N independent stateful workflow conversations. Interruption/pause-resume: not applicable today since there's no state to pause.
- **Implementation recommendation:** Note as a real product-design tension: the current single-AI-thread-per-user model is a deliberate, reasonable simplification (matches most consumer chat UX) but is incompatible with CSV 206's nested/parallel-workflow vision. Any move toward per-workflow conversation state machines would require deciding whether to keep one visible chat thread (with hidden parallel workflow states underneath) or expose multiple visible conversation threads — a product decision, not just an engineering one, and should be explicitly raised rather than assumed.

### 31. CSV 206 — Recovery, Human Override, Event-Driven Transitions & State Rules (source lines 4346–4409)

- **Understanding:** Failures never lose progress — validation errors return to the previous step, server failures resume, worker failures retry, timeouts notify. Managers can Pause/Resume/Skip/Cancel/Rollback/Restart/Transfer/Escalate any conversation, all logged. Every state transition is triggered by an explicit event (Button Click, Form Submit, Upload, Approval, API Success/Failure, Worker Completed, Reminder Timeout, Voice Command) — never implicit. Every state defines its own Required Inputs/Allowed Outputs/Business Rules/Permissions/Timeout/Retry Count/Validation Rules/"LLM Allowed?" flag/Worker Agent/UI Components. Critically: "LLM cannot store state, change state, skip state, delete state, or execute unauthorized transitions. Only suggest. Software decides."
- **Architecture/Schema implications:** An event-sourcing-adjacent pattern — transitions as discrete logged events rather than direct state mutation — which also naturally supports the audit trail and recovery requirements.
- **Gap vs. current repo:** No event log for conversation-level transitions exists (distinct from the general audit-trail infrastructure that likely exists elsewhere in this compliance-focused product for other entities). No manager override controls (pause/resume/rollback) for conversations. The "LLM cannot change state" rule is trivially satisfied today only because there IS no state for the LLM to change — once subsection 29's state fields are added, this rule needs to be actively enforced (i.e., state transitions must happen in application code after LLM calls, never be applied directly from LLM output).
- **Implementation recommendation:** When implementing subsection 29's state columns, build the transition function as a single, application-code-only mutation path (never let LLM JSON output directly `UPDATE` the state column) — this is a discipline/code-review concern as much as a schema concern, and should be called out explicitly in any spec written for this work.

### 32. CSV 206 — Governance, AI Coding Directive & Enterprise Conversation State Graph Proposal (source lines 4527–4581)

- **Understanding:** State definitions require Owner/Version/Approval Status/Effective Date/Compliance Tags/Audit History — no direct production changes without governance. The AI Coding Directive frames CSM as "an event-driven workflow orchestration service independent of the LLM" that the COS queries for current state, allowed next states, required info, applicable components, and whether LLM assistance is needed. Proposes an "Enterprise Conversation State Graph (ECSG)" — states as graph nodes connected to previous/next states, workflows, agents, permissions, knowledge objects, UI components, intents, decision rules, audit events, and learning observations, turning conversations into "navigable enterprise workflows rather than text transcripts." Closes with a named principle: "State-Driven Conversation Engineering (SDCE)" — "LLMs generate understanding; state machines generate execution."
- **Architecture/Schema implications:** ECSG is the state-machine sibling of subsection 27's Enterprise Intent Graph — same graph-modeling recommendation applies (edge table in Postgres, not a separate graph DB).
- **Gap vs. current repo:** Same governance gap pattern as subsection 13 (no owner/approval/compliance-tag fields on any conversation/state construct, since none of the underlying state tracking exists yet either). ECSG: not found, same status as EIG (subsection 27) — another concrete instance of the "no real knowledge graph exists" finding from the prior AI OS Certification audit.
- **Implementation recommendation:** Sequence last within this whole CSV 201-206 family — ECSG and EIG (subsection 27) could plausibly share one underlying graph-edge table (`entityRelationships: fromType, fromId, toType, toId, relationType, weight, metadata`) generalized across intents AND states rather than building two parallel graph systems. Recommend evaluating that consolidation explicitly when this work is eventually scoped, rather than defaulting to the document's implied 1:1 mapping of "one graph per CSV."

**Summary of the highest-value, most concrete gaps found in this range:**
1. `buildConversationHistory` in `src/lib/services/chat-service.ts` resends full history to the LLM every turn — directly contradicts CSV 201/206's core token-optimization rule and is the single most actionable near-term fix.
2. No intent-classification layer exists anywhere — this blocks nearly every other subsystem described (routing, state transitions, template selection) and should be the first real build.

6. The current one-AI-thread-per-user chat model is structurally incompatible with CSV 206's parallel/nested conversation vision — a product decision is needed before any state-machine work proceeds.

### 1. WCE Vision and Philosophy: workflow-as-conversation (source lines 4599-4644)

- **Understanding:** CSV 207 proposes that every enterprise workflow (e.g., filling forms, navigating menus) should instead be a guided conversation where the user states a goal and VERI leads them through it, contrasting "Traditional ERP" (find module → find screen → find button → fill form → submit) against "VERIDIAN" (tell VERI goal → VERI guides workflow → business completed).
- **Architecture/Schema implications:** Implies a workflow orchestration layer sitting between intent recognition and the actual business CRUD/service layer, decoupled from any specific UI screen.
- **Gap vs. current repo:** Partially there in spirit but not in mechanism. The repo has `src/app/api/ai/orchestrate/route.ts`, an event-driven orchestrator keyed on event types (`document.uploaded`, `item.overdue`, `notice.received`, `deadline.approaching`), but this is a reactive automation dispatcher, not a conversational front-end that replaces form navigation. All actual business modules (BOQ, procurement, compliance) are still conventional CRUD forms/pages under `src/app/(app)/`.
- **Implementation recommendation:** Would require a new "conversation goal → workflow resolution" layer that maps a stated user goal to one of the existing service-layer functions, then drives a guided multi-turn UI. High effort; depends on subsections 5-6 (workflow definition model) existing first.

### 2. WCE Core Architecture and Workflow Definition (source lines 4657-4722)

- **Understanding:** Defines a pipeline (User → Intent Intelligence Engine → Workflow Conversation Engine → Conversation Planner → Conversation State Machine → Worker Agents → Business Engine → Conversation Renderer → User) and a workflow decomposition of Workflow → Stages → Steps → Questions → Validations → Approvals → Execution → Completion, with a Purchase Order example (Vendor → Items → Taxes → Approval → PO Created).
- **Architecture/Schema implications:** Needs a hierarchical workflow model (workflow has many stages, stage has many steps, step has questions/validations) as first-class persisted entities, not just implicit code paths.
- **Gap vs. current repo:** Partially found. `src/lib/db/schema.ts` (~line 4636 onward) has a real "Shared Approval Workflow Engine" (Wave 51): `approvalWorkflowDefinitions`, `approvalWorkflowStepDefinitions`, `approvalWorkflowInstances`, `approvalWorkflowStepInstances`, `approvalWorkflowStepApprovals`, backed by a real service (`src/lib/services/approval-workflow-service.ts`) and routes (`src/app/api/approval-workflows/*`) — this is a genuine multi-step, condition-gated (`erpWorkflowConditionOperatorEnum`: gt/gte/lt/lte/eq), role-gated (`approverRole`) workflow engine modeled explicitly on ERPNext's Workflow/Workflow Transition doctypes per the code comment. Separately, the PMS (issue-tracking) module has its own independent state machine — `pmsWorkflowTransitions` (schema.ts ~line 2703, `fromStatusId`/`toStatusId`/`role`-gated) with its own route (`src/app/api/pms/workflow-transitions/route.ts`) — i.e., there are already *two* non-unified workflow-transition mechanisms in the repo. However, both only cover the **approval/status-transition** stage of the CSV 207 model — neither has "Stages," "Questions," or a conversational front-end, and both are invoked as backend gates on existing CRUD actions, not as the entry point that replaces navigation.
- **Implementation recommendation:** The approval-workflow tables/service are a solid foundation to extend rather than replace, and unifying the PMS-specific `pmsWorkflowTransitions` into the generic `approvalWorkflowDefinitions` engine (rather than maintaining two parallel mechanisms) is a worthwhile cleanup independent of the larger WCE build. Would then need new tables for pre-approval stages (Questions/Information Collection) and a runtime "conversation state machine" that walks a user through them before invoking the existing approval-workflow engine. Depends on subsection 1.

### 3. Workflow Object Schema and Categories (source lines 4723-4754)

- **Understanding:** Specifies a canonical `Workflow` JSON object (workflowId, name, industry, module, version, steps[], approvals[], workerAgents[], estimatedTime) and 17 workflow categories spanning Master Data through Administration.
- **Architecture/Schema implications:** A generic, industry/module-tagged workflow catalog table, likely with JSONB step/approval arrays or normalized child tables.
- **Gap vs. current repo:** Not found as a generic catalog. The repo's `approvalWorkflowDefinitions` table has `entityType`-scoped definitions (one workflow per entity type) rather than an industry/module-tagged universal catalog with `workerAgents[]` linkage.
- **Implementation recommendation:** Low priority standalone; would naturally emerge from extending subsection 2's tables with `industry`, `module`, and `workerAgents` columns.

### 4. Conversation-First Workflow Lifecycle and Progressive Collection (source lines 4755-4794)

- **Understanding:** Every workflow should follow Greeting → Goal Confirmation → Context Detection → Information Collection → Validation → Execution → Result → Next Suggestions, and information should be collected one logical step at a time (never 20 questions at once).
- **Architecture/Schema implications:** Requires a conversation-turn state tracker per active workflow instance, distinct from the instance's business data.
- **Gap vs. current repo:** Not found. No conversational multi-turn collection pattern exists; all data entry in the repo happens via standard multi-field web forms (single-page submission), not progressive one-question-at-a-time flows.
- **Implementation recommendation:** Depends heavily on subsections 1-2. Would need a new `WorkflowStepInstance`-style state row per in-progress workflow tracking which question is "current."

### 5. Predictive Workflow and Workflow Templates (source lines 4795-4822)

- **Understanding:** VERI should predict the next likely action after a step completes (e.g., after Project Created, suggest Upload Drawings → Upload BOQ → Assign Team → Kickoff Meeting) so the user only confirms rather than navigates; every workflow should ship as a reusable Template bundling conversation, validation, permissions, worker agents, business rules, UI components, analytics, and learning rules.
- **Architecture/Schema implications:** A `WorkflowTemplates` table plus a "next step prediction" rule engine (could be deterministic sequencing rules rather than ML).
- **Gap vs. current repo:** Not found. PROJEXA's construction modules (BOQ, site diary, progress) exist as independently invoked API routes/pages (`src/app/api/v1/projexa/*`) with no cross-module "what's next" prediction or template chaining between them.
- **Implementation recommendation:** Could start deterministic (a static "next suggested module" lookup table keyed by the module just completed) before any ML involvement — low effort, high visible payoff, and a natural fit with subsection 21 (Suggestion Intelligence Engine) below.

### 6. Dynamic/Adaptive Workflow by Experience Level (source lines 4823-4832)

- **Understanding:** The same workflow should render differently for a beginner (detailed guidance, many questions) vs. an experienced user (minimal questions, smart defaults) — same underlying workflow, different conversation depth.
- **Gap vs. current repo:** Not found — see subsection 15 (ACCE) for the fuller treatment; this is the same gap restated at the workflow layer specifically. No `experienceLevel` field or beginner/expert branching exists anywhere in `src/`.
- **Implementation recommendation:** Deferred to subsection 15; do not build workflow-level adaptation before the underlying user experience-level model exists.

### 7. Workflow Context Auto-Detection (source lines 4834-4844)

- **Understanding:** VERI should automatically know Current Company/Project/Client/Vendor/Financial Year/User/Department/Location from session context so it never asks the user to re-specify these — "never ask twice."
- **Architecture/Schema implications:** A session-scoped context resolver that all workflow steps read from, rather than each form re-collecting tenant/project scoping.
- **Gap vs. current repo:** Partially found. The repo is genuinely multi-tenant with `requireAuth()` (`src/lib/supabase/auth-guard.ts`) establishing the authenticated user/org per request, and most API routes already scope queries by org/company/project IDs pulled from the session or URL rather than re-asking. However, there's no unified "workflow context" object that's explicitly threaded through a conversational flow — the pattern exists at the API-auth layer, not as a reusable context-carrying object for guided workflows (which don't exist yet per subsection 1).
- **Implementation recommendation:** The auth-guard/session-scoping pattern is reusable groundwork; a future workflow engine should read from it rather than reinventing context resolution.

### 8. Workflow Components, Multi-Agent Workflow, and Workflow Memory (source lines 4846-4890)

- **Understanding:** Workflows are built from reusable components (Question → Validation → Decision Card → Approval Card → Action Card → Summary → Completion); a single workflow (e.g., Invoice) is expected to hand off between multiple specialist "worker agents" (Invoice Agent → GST Agent → Validation Agent → Approval Agent → Notification Agent → Learning Agent) without the conversation breaking; and every in-flight workflow persists its own memory (completed/skipped steps, current step, pending documents/approvals, assignee, next actions, ETA).
- **Architecture/Schema implications:** Needs a `WorkflowInstance`-level memory/state table (separate from `WorkflowHistory`) and a defined worker-agent handoff protocol.
- **Gap vs. current repo:** Not found as a runtime pattern. The repo's "worker agents" (`src/lib/ai-team/roster.ts`, ~50 roles across teams: VERIDIAN_AI_OS, ENGINEERING, QUALITY_SAFETY, LEGAL_COMPLIANCE, DATA_TEAM, SALES_MARKETING, FINANCE, HR, ADMIN, and four GUARDRAIL_* tiers) are confirmed by direct inspection to be the platform's own internal company org chart (roles like "Senior Backend Engineer," "QA Engineer," "Workflow Orchestrator," "AI Router / Task Classifier") — dispatched via OpenRouter through `src/app/api/ai/team/dispatch/route.ts` (classify via AI Router → execute via `runRole` in `src/lib/ai-team/team-service.ts` → Guardrail check), gated to `veridian_admin` only — to build/operate VERIDIAN itself, explicitly distinct (per the file's own header comment) from "customer-facing Orchestra Layers." A separate, much smaller `ai-os/boss/ROSTER.yaml` (3 entries: VEDABOSS/ZAI/SENTINEL) is a different, higher-level repo-governance roster, confirming there isn't even internal consistency in what "the roster" means across the codebase. None of the roles in either roster resemble runtime business-process specialists like "Invoice Agent," "GST Agent," or "Approval Agent" that would hand off mid-transaction. The GST computation itself exists as a deterministic engine (`src/lib/engines/`, per the 247-engine registry), invoked directly by service code — there's no agent-to-agent handoff layer around it.
- **Implementation recommendation:** This is a meaningful naming/conceptual collision worth flagging explicitly: the source doc's "worker agents" are business-transaction specialists (invoked per workflow step, largely deterministic where possible), while the repo's only existing "agent roster" is a dev-team automation concept. Building CSV 207's worker-agent model would be a new, additive layer — not an extension of `ai-team/roster.ts`. Should be scoped as its own subsystem: a lightweight dispatcher that calls existing deterministic engines/services in sequence per workflow step, escalating to LLM only where the engine doesn't cover the case.

### 9. Workflow Intelligence, Optimization, and Analytics (source lines 4891-4923)

- **Understanding:** VERI should continuously ask whether a workflow is blocked, automatable, or ready for the next stage; continuously learn the fastest/most common/least-error path and auto-improve; and track completion time, abandonment, approval time, rework, average questions/clicks, automation %, and token/LLM usage per workflow.
- **Architecture/Schema implications:** A `WorkflowAnalytics` time-series/aggregation table plus a learning-rule feedback loop.
- **Gap vs. current repo:** Not found. No workflow-level analytics table or abandonment/completion-time tracking exists (the repo does have various domain-specific analytics, e.g. construction KPIs, but nothing generic to a workflow-execution layer that doesn't itself exist yet).
- **Implementation recommendation:** Blocked on subsections 1-2 (no workflow runtime to instrument yet). Analytics schema should be designed alongside the runtime, not bolted on after.

### 10. Workflow Decision Matrix and Recovery (source lines 4924-4954)

- **Understanding:** For every workflow step, the system should pre-determine at design time whether it's handled by software, a worker agent, a business rule, a DB call, an API, an LLM, or a human approval — "everything determined before execution," which is a strong statement against runtime LLM improvisation. Separately, workflows must be resumable after logout/browser-close/network/server failure or device switch, with exact state restoration.
- **Architecture/Schema implications:** A per-step "execution mode" enum/config resolved at workflow-definition time, and a durable, device-agnostic workflow-instance state store (the `WorkflowRecovery` table named in subsection 24).
- **Gap vs. current repo:** Not found. There's no workflow runtime to recover, so no recovery mechanism exists. Note this principle ("decide software vs. LLM before execution, not at runtime") is consistent with how the repo already treats the 41-implemented deterministic engines (`src/lib/engines/`) versus AI orchestration routes — the *engine registry itself* embodies this decision-matrix philosophy at the computation level, just not yet at the workflow-step level.
- **Implementation recommendation:** When the workflow runtime is eventually built, the existing engine-registry categorization (deterministic-first) should be reused as the "Decision Matrix" input rather than inventing a parallel classification.

### 11. Human Override and Workflow Governance (source lines 4945-4967)

- **Understanding:** Managers must be able to pause/resume/skip/rollback/reassign/escalate/cancel any in-flight workflow with full logging; every workflow definition needs an owner, department, version, reviewer, approval, audit trail, compliance tag, KPIs, and policies attached.
- **Architecture/Schema implications:** Workflow-instance-level admin actions (state-mutating, audit-logged) and workflow-definition-level governance metadata columns.
- **Gap vs. current repo:** Partially found at the approval-workflow layer only. `approvalWorkflowStepApprovals` supports approve/reject decisions with presumable audit fields, and the repo generally has strong audit-logging conventions elsewhere (e.g., compliance module). But pause/skip/rollback/reassign primitives for a broader conversational workflow don't exist because the workflow runtime itself doesn't exist yet.
- **Implementation recommendation:** Governance metadata (owner, version, reviewer) should be added to any new `WorkflowDefinitions`-equivalent table from day one; retrofitting governance columns later is expensive.

### 12. Workflow Database Design and Worker Agents List (source lines 4968-4997)

- **Understanding:** Prescribes 15 primary tables (WorkflowDefinitions, WorkflowStages, WorkflowSteps, WorkflowRules, WorkflowTemplates, WorkflowAnalytics, WorkflowHistory, WorkflowAssignments, WorkflowPermissions, WorkflowVersions, WorkflowEvents, WorkflowRecovery, WorkflowLearning, WorkflowRecommendations, WorkflowPolicies) and 10 named worker-agent roles specific to workflow management itself (Workflow Planner Agent, Workflow Optimizer, Workflow Recovery Agent, Workflow Learning Agent, Workflow Analytics Agent, Workflow Validator, Workflow Recommendation Agent, Workflow Governance Agent, Workflow Automation Agent, Workflow Simulation Agent).
- **Architecture/Schema implications:** This is the concrete schema checklist for subsections 1-11 combined.
- **Gap vs. current repo:** Only the approval-adjacent subset exists (`approvalWorkflowDefinitions`, `approvalWorkflowStepDefinitions`, `approvalWorkflowInstances`, `approvalWorkflowStepInstances`, `approvalWorkflowStepApprovals` — roughly 5 of the 15 named tables, and only covering the approval slice, not stages/templates/analytics/recovery/learning/recommendations/policies). None of the 10 workflow-specific worker-agent roles exist under any name in `src/lib/ai-team/roster.ts` or elsewhere.
- **Implementation recommendation:** Treat this as the master checklist when scoping a real WCE build; roughly two-thirds of the schema and all of the named agents are greenfield work.

### 13. Token Optimization via ID-Only References (source lines 4998-5012)

- **Understanding:** Never send the full workflow object to the LLM/client on each turn — send only `WorkflowID`, `CurrentStepID`, `StageID`, `StateID`, `ContextID`, and let the software reconstruct full context server-side, avoiding repeated prompt bloat.
- **Architecture/Schema implications:** Implies a clean separation between a compact "state pointer" passed over the wire and the full workflow object hydrated server-side per request.
- **Gap vs. current repo:** Not directly applicable yet (no workflow runtime), but the *principle* is already practiced elsewhere in the repo's AI-cost-consciousness — e.g., the token-usage ledger and the documented root-cause fix where full conversation-history-growth (resending large tool-call histories every iteration) caused a $11.44-of-$12.34 billing spike in the AI Workforce dispatcher, per `src/lib/ai-team/roster.ts`'s header comment. That incident is direct evidence the team already understands and has been burned by NOT following this exact principle in an adjacent system.
- **Implementation recommendation:** When building the workflow runtime, treat this ID-only pattern as non-negotiable given the team's own prior cost incident; would pair naturally with the token-usage ledger already in place.

### 14. Workflow Graph, AI Integration Rule, and KPIs (source lines 5013-5066)

- **Understanding:** Complex workflows should be modeled as graphs (branch/parallel/merge) rather than linear step chains; the LLM should be reserved strictly for reasoning, summarization, document understanding, risk assessment, and creative writing — "everything else: software"; and 10 KPIs are defined (completion %, avg time, automation %, prediction accuracy, avg clicks/typing, token usage, LLM calls, recovery rate, satisfaction).
- **Gap vs. current repo:** Not found as a graph model (no branching/parallel workflow representation anywhere). The "LLM only for reasoning" rule is philosophically consistent with the repo's deterministic-engines-first posture (`src/lib/engines/`, 41 of 247 implemented) but isn't formally enforced as a platform rule — individual services just happen to call engines directly rather than LLMs for calculations.
- **Implementation recommendation:** Formalizing "LLM only for reasoning" as an enforced platform rule (e.g., a lint rule or code-review checklist item) would be low-cost and consistent with existing practice; the workflow-graph model itself is deferred until the workflow runtime exists.

### 15. AI Coding Implementation Directive for WCE (source lines 5067-5080)

- **Understanding:** The directive instructs that business modules define workflows *declaratively* via metadata (stages, steps, validations, permissions, worker agents, decision points, components) and the engine executes these dynamically rather than hardcoding UI logic — supporting multi-step, branching, parallel, pause/resume, approvals, agent collaboration, and being invocable identically via conversation, UI, API, automation, or voice without duplicating business logic.
- **Architecture/Schema implications:** This is the strongest architectural mandate in the section: a metadata-driven interpreter pattern, where business logic is defined once and rendered/executed through multiple channels.
- **Gap vs. current repo:** Mostly not found. The repo's PROJEXA modules and compliance workflows are hardcoded per-module (each has its own API route, page, and service file) rather than metadata-driven declarations interpreted by a shared engine — this is the opposite of the directive. Concretely, PROJEXA's construction modules (BOQ, work-progress, site diary, RFIs, submittals, punch-list, change-orders — schema.ts ~lines 7202-7520) each use their own single-column status enum (e.g., `constructionPunchStatusEnum: ['open','ready_for_review','verified_closed']`, `constructionChangeOrderStatusEnum: ['draft','pending_approval','approved','rejected']`) with transitions hand-rolled inline in service functions (e.g., `reviewSubmittal()` in the construction field-workflow service checks a hardcoded valid-status array) rather than being registered against the generic `approvalWorkflowDefinitions` engine from subsection 2 — so even the two real workflow mechanisms that do exist in the repo (`approvalWorkflowDefinitions` and `pmsWorkflowTransitions`) aren't reused by the newest modules, let alone driven by a shared declarative interpreter. The repo does have a `/api/v1/*` aliasing surface for PROJEXA that exposes the same underlying data through a differently-namespaced API, which is a partial analog to "same logic, multiple channels" but only at the API-shape level, not at the workflow-execution level.
- **Implementation recommendation:** This is the single highest-leverage, highest-effort item in this document range. A genuine implementation would require introducing a workflow-definition DSL/schema (YAML or JSON, consistent with the `ai-os/specs/SPEC-*.yaml` convention already used elsewhere in the repo) and a generic interpreter service — a multi-wave project, not a single PR. Should be sequenced after core business modules stabilize, since it's a meta-layer over existing services rather than new business capability.

### 16. Enterprise Workflow Graph (EWG) Proposal (source lines 5082-5100)

- **Understanding:** Proposes modeling the whole enterprise as a single graph whose nodes are workflows, stages, steps, worker agents, conversation states, intents, business rules, APIs, data models, documents, approvals, events, and KPIs, with edges representing dependencies/prerequisites/parallelism/escalation/automation opportunities — enabling VERI to reason holistically (bottleneck detection, process improvement suggestions, impact prediction, and composing new workflows from existing pieces).
- **Architecture/Schema implications:** This is effectively a proposal for a knowledge-graph-like structure scoped specifically to workflows (a precursor/sibling to the general knowledge graph discussed in later Study sections).
- **Gap vs. current repo:** Confirmed not found — a direct repo-wide search for "knowledge graph" / "KnowledgeGraph" / "knowledgeGraph" returns zero matches anywhere in `src/`. This directly corroborates the prior internal "AI OS Certification" audit's finding that a real knowledge graph does not exist despite being referenced conceptually. Notably, `ai-os/boss/BOARD.yaml` contains a task entry `{ id: AIOS-003, title: "Populate ARTIFACTS.yaml knowledge graphs", status: completed }` — marked **completed** — yet no `ARTIFACTS.yaml` file exists anywhere in the repo at all, and no graph-node/edge tables exist in the schema. This is a concrete, named instance of a false "done" claim in the platform's own task tracker, not just an absence — worth flagging explicitly since it's the kind of claim that could mislead future planning if trusted at face value.
- **Implementation recommendation:** Do not attempt EWG before the base workflow runtime (subsections 1-15) exists — a graph over workflows that don't yet exist as first-class objects is premature. This should be flagged as a downstream/Phase-2 item once WCE ships.

### 17. Workflow-First Architecture Principle (source lines 5102-5116)

- **Understanding:** States the platform-wide rule "every feature in VERIDIAN must exist as a workflow before it exists as a screen" — engineers should ask about business outcome, workflow, conversations, participating agents, decisions, approvals, automation opportunity, and token minimization *before* any UI is built, positioned as VERIDIAN's defining architectural differentiator from both traditional ERPs and generic AI assistants.
- **Gap vs. current repo:** Not followed today, and this is worth stating plainly: essentially every module found in the repo (BOQ, punch-list, submittals, RFIs, compliance checklists, inventory) was built screen-first / CRUD-first, with UI and API routes as the primary artifact and no antecedent workflow definition. This isn't a criticism of what's shipped — it's a real, current architectural gap relative to this stated principle.
- **Implementation recommendation:** This is a governing philosophy, not a buildable unit — it should inform how future modules are scoped once WCE exists, but retrofitting it onto already-shipped CRUD modules would be high-cost/low-value. Best framed as "new modules from here forward should define a workflow object first," not a rewrite mandate.

### 18. CXE Vision, Philosophy, and Design Principles (source lines 5118-5175)

- **Understanding:** CSV 208 reframes the UI layer as a "Conversation Experience Engine" — proactive rather than reactive (software observes/predicts/suggests/prepares/asks/executes, user approves), where every screen should implicitly answer "what's happening / what's next / why is this recommended / what's the impact / can I one-click this / what needs approval / what can be automated."
- **Gap vs. current repo:** Not found as a general pattern. The repo's UI (`src/components/AppSidebar.tsx`, dashboard pages under `src/app/(app)/`) is conventional dashboard/CRUD UI — informational and navigational, not proactive/predictive. No component surfaces "why is VERI recommending this" as a first-class UI pattern.
- **Implementation recommendation:** Depends on subsections 21-31 below (Suggestion Intelligence Engine) existing as a data source before any "proactive" UI can be built — UI can't predict what the backend doesn't compute.

### 19. Experience Objects and Card Taxonomy (source lines 5218-5236)

- **Understanding:** Everything shown to the user should be a structured "Experience Object" rendered as one of 16 named card types (Conversation, Approval, Suggestion, Risk, Action, Summary, Progress, Comparison, Notification, Learning, Dashboard, Insight, Timeline, Checklist, Document, Decision Card) — the UI never renders raw LLM text directly, only structured objects.

## Study 17 — CSV 212: Cognitive Learning & Evolution Engine (CLEE)

### Vision, philosophy, and objectives of CLEE (source lines 7052-7099)

The philosophy is "observe everything, learn only what matters, improve continuously, never compromise governance" — learning is a means to measurable outcomes, not an end in itself.
- **Architecture/Schema implications:** Implies a platform-wide event-capture layer distinct from any single business module, feeding a dedicated learning subsystem rather than learning being embedded ad hoc in each feature.
- **Gap vs. current repo:** No dedicated "learning engine" service exists. ### Core architecture pipeline: Enterprise Events → Learning Queue → Governance → Platform Evolution (lines 7100-7135)

- **Gap vs. current repo:** Real, partial infrastructure exists and is genuinely relevant here: `loopExecutions`, `loopImprovements`, and `loopHealthMetrics` tables in `src/lib/db/schema.ts` (~lines 1094-1132), described in a code comment as "Self-Improvement Loops + Knowledge Flow (Wave 5)." `loopExecutions` captures `observationData`/`analysisResult`/`actionTaken`/`measurementResult` per run — a near-exact match to the source's Observe→Analyze→Generate Improvement pipeline shape. ### Learning pipeline, observation engine, and pattern detection (lines 7177-7248)
- **Understanding:** A 12-stage pipeline (Observe→Capture→Normalize→Validate→Cluster→Analyze→Generate Improvement→Human Review→Approve→Deploy→Measure→Learn Again) with an Observation Engine that watches user behavior, workflow delays, navigation, errors, repeated questions, and token usage, and a Pattern Detection layer that specifically hunts for *repeated* mistakes/approvals/delays/conversations/corrections/workflows/reports/calculations.

## Study 19 — CSV 214: Enterprise Execution Orchestration Engine (EEOE)

### Vision: execution as coordinated, not merely "running code" (lines 8005-8058)
- **Understanding:** EEOE is framed as the "COO" of the Cognitive AI OS — the gap being that most AI systems (ChatGPT, Claude, Copilot per the source's own comparison) create plans but don't execute them. ### Execution Object, sources, and the 10-stage Execution Pipeline (lines 8095-8177)

- **Implementation recommendation:** Formalizing the existing implicit "read-only auto, write-actions human-gated" behavior into an explicit `executionMode` field would be a natural, low-risk first step that surfaces existing behavior rather than changing it.

### Multi-Agent Orchestration and Worker Coordination Protocol (WCP) (lines 8203-8216, 8425-8439)
- **Understanding:** A single workflow may involve many specialist agents (Planner, Validation, Document, GST, Accounting, Project, Compliance, Notification, Analytics, Learning, Coordinator) collaborating under the orchestration engine. Later, the document proposes a formal Worker Coordination Protocol (WCP) where agents exchange *structured* messages (Task Assignment, Task Accepted, Task Started, Progress Update, Dependency Waiting, Escalation Request, Completion, Failure, Retry Request, Knowledge Contribution) instead of free-form prompts, explicitly to reduce token consumption and improve reliability.

## Study 25 / CSV 220 — Meta Intelligence & AI OS Orchestrator (MIAO)

### Vision and philosophy (source lines 10693-10727)
- **Understanding:** MIAO is positioned as the "engine above all engines" — the supervisory layer that decides, for every request, which engine/worker/model/deterministic service should handle it, and whether an LLM is needed at all. Its core philosophy is a strict escalation ladder: don't ask AI to do what software can do; don't ask software to do what rules can do; don't ask rules to do what calculators can do; don't ask calculators to do what databases can do. Only call an LLM for genuine reasoning, ambiguity resolution, or natural-language understanding.
- **Architecture/Schema implications:** This is explicitly meant to be a *central runtime coordinator* — every engine and every LLM call routes through it, not a peer engine among equals.
- **Gap vs. current repo:** This philosophy is real and demonstrably practiced throughout the repo (the 247-engine deterministic registry, the GST engine's stated principle, `construction-prediction-service.ts`'s explicit rejection of ML in favor of deterministic averaging, `fde-service.ts`'s embedding-search-before-LLM-call pattern). However, there is **no single central orchestrator enforcing it**. Each service independently implements the "prefer deterministic, fall back to LLM" pattern on its own — it is a convention followed by discipline/code-review, not a structural guarantee enforced by one chokepoint. `orchestra-model-resolver.ts` resolves *which model/provider* to use per "Orchestra Layer" (a real, working per-layer model-selection abstraction, with `resolveModelConfig()` and platform/customer/BYO-key resolution) but this answers "which model" once a layer has already decided an LLM call is needed — it does not answer "should this be software/SQL/rules/calculator/LLM" the way MIAO's Intelligence Routing (§6-7) describes. That upstream classification decision is made ad hoc, per-service, by whoever wrote that service.
- **Implementation recommendation:** This is the single highest-value, most honestly-gapped concept in this entire document section relative to what's built. The repo has all the *ingredients* MIAO would coordinate (deterministic engines, worker agents, orchestra-layer model resolution, embedding-based capability matching) but no actual coordinator making the "which mechanism should handle this" decision centrally. A real implementation would NOT need to rebuild any of those ingredients — it would need a thin routing/classification layer in front of them. This is a large, cross-cutting architectural change (touches how every service currently decides to call an LLM) and should be scoped as its own dedicated initiative, likely the direct implementation target that follows this whole CSV 201-222 document series.

### Meta architecture, managed engines, and intelligence routing (source lines 10742-10828)
- **Understanding:** Lays out the full request-flow diagram (User → Conversation OS → Intent Engine → Meta Orchestrator → Decision Router → Execution Planner → AI/Software Selection → Worker Agent Network → Modules → Knowledge Graph → Learning → Analytics → Response) and lists 19 "Managed Engines" that should all report to the Meta Orchestrator (Conversation, Intent, Memory, Knowledge, Reasoning, Decision, Planning, Execution, Learning, Prediction, Innovation, Command Center, Security, Workflow, Analytics, Reporting, Notification, Integration, Governance). Every request should classify into one of 11 execution modes (Pure Software, Business Rule Engine, Calculation Engine, SQL Query, Search Engine, Knowledge Graph Query, Workflow Engine, Worker Agent, Multi-Agent Collaboration, LLM Reasoning, Hybrid).
- **Architecture/Schema implications:** This is effectively a request-classification/dispatch layer — conceptually similar to an API gateway or a rules-based router, but for cognitive tasks rather than HTTP routes.
- **Gap vs. current repo:** Of the 19 "Managed Engines" listed, only a handful have real implementations in this repo (Workflow-ish: `automation-rule-service.ts`; Analytics: `analytics-engine.ts`; Security: `security-engine.ts`; Governance-ish: `policy-enforcement-engine.ts` per `fde-service.ts`'s import). Named engines like "Command Center," "Intent Engine," "Memory Engine" as distinct architectural layers were not confirmed to exist as such (assistant-memory-service.ts covers some "Memory Engine" territory). There is no unifying registry of these as peer "engines reporting to a Meta Orchestrator" — they exist as independent services with independent conventions.
- **Implementation recommendation:** Before building a literal MIAO, an inventory/registry step is prerequisite: catalog which of the 19 named engines have real analogues in the repo today (several do, informally) versus which are purely conceptual. ### AI Invocation Policy and Enterprise Cognitive Loop (source lines 10830-10895)
- **Understanding:** Formalizes the escalation ladder as an explicit evaluation sequence (software? → SQL? → calculator? → rule engine? → workflow? → worker agent? → knowledge graph? → only then LLM) and defines an 11-step "Enterprise Cognitive Loop" (Observe → Understand → Classify → Route → Execute → Validate → Measure → Learn → Optimize → Update Knowledge → Improve → Repeat) as "the heartbeat of VERIDIAN."

## Study 26 / CSV 222 — Functional Development Engineering (FDE) & Capability Evolution Engine

### Vision and philosophy (source lines 11201-11231)
- **Understanding:** FDE reframes how VERIDIAN should respond to new customer requirements: instead of treating every request as a new feature to code, VERIDIAN should first determine whether the requested functionality already exists, can be assembled from existing capabilities, or truly requires new development. The stated goal: "every customer investment should improve the platform for every future customer." Philosophically: customers request outcomes, developers think in features, VERIDIAN should think in capabilities — never asking "how do I code this" before first asking "do I already know how to do this."
- **Architecture/Schema implications:** Implies a capability-search step must precede any code-generation step, for any AI-driven development flow.
- **Gap vs. current repo:** **This is the one CSV in this entire Part that is already substantially, genuinely implemented — not a concept gap but a real, working system.** `src/lib/services/fde-service.ts` (labeled internally as "Wave 42 (VERI FDE — Forward Deployed AI, PLATFORM_STRATEGY.md §23)") implements exactly this philosophy: `submitFdeRequest()` runs an embedding-based `findSimilarCapabilities()` search first; a high-confidence match (`HIGH_CONFIDENCE_THRESHOLD = 0.9`) answers instantly with **zero LLM calls**; only uncertain matches escalate to an LLM evaluating a small top-K candidate set (`CANDIDATE_LIMIT = 8`), not the org's entire capability roster. There is a real `/fde` UI page ("Request a capability" button) and a `fde_requests` table (`schema.ts:3261`).
- **Implementation recommendation:** No new build needed for the core concept — it exists and is live. See later subsections for the specific gaps between this real implementation and the CSV's fuller vision (10-level reuse hierarchy, Capability DNA, marketplace).

### Core architecture pipeline (source lines 11246-11299)
- **Understanding:** User Requirement → Conversation Engine → Intent Understanding → Functional Requirement Analyzer → Capability Discovery Engine → Reuse Analyzer → Gap Analysis Engine → FDE Engine → Implementation Factory → Testing → Governance Review → Capability Registry → Global Worker Library → Future Reuse.
- **Gap vs. current repo:** The real `fde-service.ts` pipeline is a compressed version of this: (embedding search) → (high-confidence auto-answer OR LLM evaluation of top-K) → (propose new Worker Agent via `proposeWorkerAgent()` if no match) → (human approval, per the FDE header comment: "role-gated, human-approval-gated pipeline"). Missing relative to the CSV's fuller pipeline: no distinct "Functional Requirement Analyzer" stage, no formal "Gap Analysis Engine" producing a document (see next subsection), no "Implementation Factory" or "Testing" automation stage — new capabilities today are proposed as Worker Agent definitions (prompt templates + input/output schema), not code, so there's no code-generation/testing step to formalize in the same sense the CSV envisions.
- **Implementation recommendation:** The compression is reasonable and arguably correct for this platform's actual capability model (agents defined by prompt + schema, not by generated source code) — a literal "Implementation Factory" that writes and tests new TypeScript would be a much larger, separate initiative (closer to the AI Dev Team's `/api/ai/team/dispatch`, which does generate and PR real code, per prior context) than an extension of FDE itself.

### Functional requirement understanding and capability discovery (source lines 11301-11335)
- **Understanding:** The engine should understand business objective, industry, workflow, constraints, compliance, UI expectations, data model, permissions, and worker agents needed — without the user describing implementation details. Before creating anything, it should search across a long list of registries: Capability Registry, Module Library, Feature Library, Workflow Library, Worker Agent Library, Conversation Library, Knowledge Graph, Calculation Engine, API Registry, Integration Registry, Prompt Library, Template Library, UI Component Library.
- **Gap vs. current repo:** The real implementation searches exactly 4 entity types (`CAPABILITY_ENTITY_TYPES = ["worker_agent", "automation_rule", "module", "prompt_pattern"]`), not the full ~13-registry list the CSV describes. Notably absent from the real search scope: Calculation Engine (the 41-implemented-of-247 deterministic engines in `src/lib/engines/` are not embedded/searchable via FDE), API Registry, Integration Registry, UI Component Library, Template Library.
- **Implementation recommendation:** Directly actionable, low-risk extension: widen `CAPABILITY_ENTITY_TYPES` and add corresponding `indexCapability()` calls for calculation engines (so a request like "I need a GST split calculator" would surface the existing `gst-engine.ts` functions instead of risking a duplicate proposal). This is the same recommendation made under Study 23's Marketplace subsection — the two CSVs converge on the same concrete next step.

### Capability Reuse Hierarchy (source lines 11337-11358)
- **Understanding:** Defines a strict 10-level escalation order before new development is allowed: Configuration → Existing Workflow → Existing Worker Agent → Existing Module → Existing API → Existing Integration → Existing Capability Composition → Capability Extension → New Capability → New Platform Module. New development is explicitly the last resort.
- **Gap vs. current repo:** The real `fde-service.ts` implements a much flatter version: essentially levels 3 and 9 only (existing Worker Agent match, or propose a brand new Worker Agent) — there's no attempt at levels 1-2 (pure configuration, existing workflow reuse) or 4-8 (module/API/integration/composition/extension) as distinct, ordered checks. A request that could be satisfied by *composing* two existing worker agents, for instance, is not distinguished from one requiring a wholly new agent — `findSimilarCapabilities` returns single best matches, not composition proposals.
- **Implementation recommendation:** This is the most conceptually valuable and least-implemented part of CSV 222 relative to what exists. A full 10-level hierarchy is a large undertaking, but even adding one intermediate level — checking whether an existing `automation_rule` (already one of the 4 searched entity types) could be *configured* differently to satisfy the request, before falling through to "propose new agent" — would meaningfully close the gap without a full rebuild. Should be scoped as an explicit follow-on to the existing FDE service rather than a new system.

### Functional Gap Analysis (source lines 11360-11374)
- **Understanding:** When reuse genuinely isn't possible, the engine should produce a structured Gap Analysis document (problem definition, business context, current limitation, missing capability, expected value, estimated reusability, affected modules, suggested architecture, dependencies, security considerations, token cost impact, governance requirements) that becomes part of engineering documentation.
- **Gap vs. current repo:** Not implemented as a distinct artifact. When `fde-service.ts` finds no match, it goes straight to proposing a new Worker Agent (name/domain/description/promptTemplate/inputSchema/outputSchema) via `proposeWorkerAgent()` — this is a *design*, not a *gap analysis*. There's no separate "why doesn't this exist yet, and what's the estimated cost/architecture" document produced or stored.
- **Implementation recommendation:** Could be added as an optional richer `proposal` object in the existing `FdeEvaluation` type (already has a `proposal` field) — extending it with `expectedValue`, `estimatedReusability`, and `tokenCostImpact` fields would move the existing system closer to this subsection's spec without new infrastructure.

### Capability classification and composition (source lines 11376-11415)
- **Understanding:** Every new capability should be tagged along 13 dimensions (industry, business domain, workflow, module, feature type, worker type, integration type, UI component, data model, permission model, knowledge pack, conversation type, automation category) for searchability. Many requests should be fulfilled by *composing* existing capabilities rather than building new ones — worked example: "Digital Inspection App = Forms + GPS + Camera + OCR + Workflow + Approval + Reports + Notification + Project Module."
- **Gap vs. current repo:** The real `workerAgents` table has a `domain` field (used in `buildCapabilityContent()`) but not the full 13-dimension taxonomy. Composition is not implemented — `findSimilarCapabilities()` returns ranked single matches by embedding similarity, not decomposed multi-capability bundles.
- **Implementation recommendation:** Composition is a meaningfully harder problem than single-match search (it requires recognizing that a request decomposes into N sub-capabilities, not just finding the single closest one) — this is a genuine research/design gap, not just missing plumbing. Lower priority than the Reuse Hierarchy gap above, since composition presupposes a working multi-level reuse check exists first.

### Autonomous Development Pipeline and Worker Agent Evolution (source lines 11417-11450)
- **Understanding:** When a genuine gap exists, the standardized pipeline is Requirement → Design → Architecture → Data Model → API Design → Worker Agent Design → UI Components → Business Rules → Tests → Documentation → Deployment Proposal → Capability Registration. For capabilities requiring specialized intelligence, generate a new Worker Agent or enhance an existing one, auto-registered into the Global Worker Registry.
- **Gap vs. current repo:** `proposeWorkerAgent()` (in `worker-agent-service.ts`, called from `fde-service.ts`) covers roughly the "Worker Agent Design" and "Capability Registration" steps of this pipeline directly — a proposed agent is a real row in the `workerAgents` table with `lifecycleStatus: "proposed"`, later presumably approved into `"approved"`/`"published"` status (per the `listPublished...` filter defaulting to `["approved", "published"]`). The earlier pipeline stages (Architecture, Data Model, API Design, UI Components, Tests) are not part of this flow at all — because, again, the platform's unit of new capability is a Worker Agent (prompt + schema), not generated application code with its own data model/API/UI.
- **Implementation recommendation:** No gap to close here beyond what's already noted — the mismatch is structural (CSV 222 assumes code-generation-level capabilities; the real system's capabilities are prompt/schema-level) rather than a missing feature.

### Global Capability Registry (source lines 11452-11469)
- **Understanding:** Every approved capability should store Capability ID, purpose, business domain, inputs, outputs, dependencies, owner, version, documentation, worker agents, token cost, quality score, usage statistics, compatibility, and lifecycle — described as becoming "the memory of the platform."
- **Gap vs. current repo:** Partially implemented. `capability-registry-service.ts`'s `buildCapabilityContent()` captures name/domain/description/inputSchema/outputSchema as embeddable text (purpose, domain, inputs, outputs — 4 of the ~15 listed fields). The `workerAgents` table has `lifecycleStatus`. Not present: version numbers, token cost per capability, quality score, usage statistics (how often has this capability actually been matched/reused by FDE?), or compatibility metadata.
- **Implementation recommendation:** "Usage statistics" is the highest-value missing field — without tracking how often `findSimilarCapabilities()` actually surfaces and uses a given capability, there's no feedback signal to know which capabilities are valuable versus dead weight (directly relevant to Study 23's "Continuous Capability Evolution" subsection too, which asks for the same usage-frequency tracking). This would require adding a counter/log increment at the point `submitFdeRequest()` accepts a high-confidence match — a small, targeted addition to existing code (not touched in this study-only pass, per instructions).

### Capability Marketplace and Continuous Capability Evolution (source lines 11471-11498)
- **Understanding:** Every product should be able to consume Worker Agents, Modules, Features, Dashboards, Templates, Workflows, Integrations, Reports, Calculators, Knowledge/Industry Packs as installable, configuration-only assets. Usage frequency, feedback, performance, errors, business value, token cost, maintenance cost, and reuse count should be tracked so frequently-used capabilities receive improvement proposals.
- **Gap vs. current repo:** The `/api/v1/projexa/*` aliasing namespace (per prior session context: PROJEXA exposed via VERIDIAN's API without a rebuild) is arguably a real, working instance of "install by configuration, not by rebuilding" — PROJEXA reuses VERIDIAN's underlying capabilities through API aliasing rather than duplicating them. This is a genuine example of the marketplace philosophy already succeeding at the product level, even without a literal "marketplace" catalog UI.
- **Implementation recommendation:** Worth noting explicitly in any gap-analysis rollup: the PROJEXA-on-VERIDIAN pattern is proof the underlying philosophy works at least once in practice, even though no generic, browsable marketplace UI/catalog exists for it.

### AI Coding Rules for capability-first development (source lines 11500-11520)
- **Understanding:** Before generating code, the AI must verify: can this be configured? composed? extended? solved by metadata? handled by an existing worker or module? Only if all answers are "no" may code generation begin. Human approval is required for new modules, new platform capabilities, breaking changes, security changes, public APIs, schema changes, and global worker registration.
- **Gap vs. current repo:** The "code-generation last resort" check is real for the Worker Agent creation path specifically (FDE checks for a match before `proposeWorkerAgent()` is ever called) — but the separate, real "AI Dev Team" system (`/api/ai/team/dispatch`, 27-role org chart on OpenRouter, per prior context) generates actual source code and opens PRs; it's not confirmed whether that pipeline runs an equivalent "does this already exist" check before generating code, or whether it's a separate code path with no capability-discovery gate. The human-approval gates for schema changes/breaking changes/public APIs are enforced structurally by this repo's own branch-protection rule (per `AGENTS.md`: all changes go through a PR requiring passing CI, `enforce_admins` on, no direct pushes to `main`) — a different but functionally overlapping governance mechanism from what CSV 222 describes.
- **Implementation recommendation:** Worth explicitly checking (in a future, non-study session) whether the AI Dev Team's code-dispatch pipeline calls `findSimilarCapabilities()`/FDE before generating new code — if it doesn't, that's a real duplicate-work risk: two separate "propose something new" pipelines (FDE's worker-agent proposals and the AI Dev Team's code generation) that don't share a reuse-check gate.

### KPIs, AI Coding Implementation Directive, and Capability DNA (source lines 11522-11581)
- **Understanding:** KPIs include capability reuse rate, new capability creation rate, average development time, reuse savings, token savings, marketplace adoption. The implementation directive reiterates the Discovery → Reuse → Gap Analysis → Architecture → Design → Implementation → Testing → Governance → Registration → Marketplace Publication sequence. "Capability DNA" proposes a permanent record per capability (functional purpose, business intent, domain, inputs/outputs, APIs, UI components, worker agents, data models, business rules, events, permissions, dependencies, token profile, reuse score, evolution history, compatibility matrix) enabling search by *functional similarity* rather than name — the CSV's own example: recognizing a "Site Inspection Checklist" and a "Quality Audit Checklist" share 90% of their underlying capability and should reuse the same implementation.
- **Architecture/Schema implications:** "Capability DNA" searchable by functional similarity, not name, is precisely what embedding-based search already does — this is not a new concept relative to the existing implementation, just a more elaborate metadata schema around it.
- **Gap vs. current repo:** The core mechanism this subsection asks for — functional-similarity search instead of name-matching — **is already exactly what `findSimilarCapabilities()` does** (semantic embedding search over `buildCapabilityContent()`, which explicitly includes input/output schema "so the vector captures the contract, not just the prose description," per that function's own header comment — literally written to solve the "Site Inspection Checklist vs Quality Audit Checklist" problem this CSV subsection describes). What's missing is only the fuller "Capability DNA" metadata record (reuse score, evolution history, compatibility matrix, token profile) sitting alongside that working search mechanism — not the search mechanism itself.
- **Implementation recommendation:** This is worth flagging prominently: the CSV's most sophisticated-sounding idea (DNA-based functional matching) is already solved in the real repo by a simpler, working mechanism (embedding search over a contract-aware content string). No new search technology is needed — only richer bookkeeping fields on top of what exists.

### Closing engineering principle and Capability Lifecycle Engineering (source lines 11583-11631)

- **Gap vs. current repo:** Most of the CLE lifecycle stages map onto real, working pieces already covered in prior subsections (Search = `findSimilarCapabilities`, Reuse Analysis = the `HIGH_CONFIDENCE_THRESHOLD` check, Design = `proposeWorkerAgent`'s proposal shape, Deployment/Registration = `lifecycleStatus` transitions). Usage Analytics and Deprecation are the two stages with no confirmed implementation.
- **Implementation recommendation:** Given how much of CLE already exists in working form, the two genuinely missing pieces (usage-count tracking per capability, and a formal deprecation/replacement status + workflow) are the concrete, scoped, low-risk next steps for this whole CSV 222 area — both were already independently identified above (Global Capability Registry subsection) and are reinforced here as the natural closing recommendation for this Part.

## Study 28 — Human-Centric UI/UX Constitution

### 28.1 Vision & Core Philosophy (lines 11632–11722)
- **Understanding:** Frames VERIDIAN as neither a chatbot nor a traditional ERP but a unified AI-native OS where AI, software, automation, and human judgment share one workspace. States four philosophy pillars: Human First (every feature must reduce effort), Context Before Conversation (context precedes any chat), Recognition over Recall (present choices, don't demand typed recall), and Progressive Guidance (reveal one decision at a time). Closes with a six-step universal interaction loop: Understand → Guide → Confirm → Execute → Learn → Improve.
- **Architecture/Schema implications:** "Context Before Conversation" implies every conversation/message row must carry a foreign key to a business-context object (product/project/module/customer/etc.) established *before* the row is created — i.e., context is not inferred post-hoc from message text.
- **Gap vs. current repo:** Partially present. `src/components/veri-chat/veri-chat-context.tsx` and `VeriComposer.tsx` implement a real capability-tree-driven selection flow (`CapabilityNode`, `PathSegment`) that must be walked before a task/conversation is created — this is a genuine, working instance of "context before conversation," not vaporware. - **Implementation recommendation:** No new build required for the core mechanism; recommend documenting the existing capability-tree flow explicitly as the "Human Intent Engine" pipeline (naming alignment with Study 29) and auditing whether a "Learn/Improve" feedback loop (usage-based reordering of tree nodes) exists — initial read suggests it does not.

### 28.2 The Five Sections — Left Navigation Panel (lines 11723–11763)
- **Understanding:** Specifies a collapsible left nav containing standard enterprise modules (Dashboard, Finance, Sales, CRM, HR, etc.) plus org-specific custom modules. It must remember recently-opened/favourite/frequent modules and have VERI proactively recommend modules based on current work.
- **Architecture/Schema implications:** Requires a per-user `nav_preferences` or `recent_modules` table (module_id, user_id, last_opened_at, pin_order) and a recommendation hook keyed off current task context.
- **Gap vs. current repo:** A left sidebar exists (`src/components/ui/sidebar.tsx` is the shadcn primitive; an `AppSidebar`-style implementation is referenced in `CLAUDE.md`'s structure notes). Standard module links likely exist given the breadth of `src/app/(app)/`. Not verified: persistence of "recently opened" / "favourites" or AI-driven module recommendations — no `recent_modules` or `nav_preferences` table found during this pass in the schema areas checked.

### 28.3 Main Workspace — dual Software/Conversation modes (lines 11764–11808)

Remaining work: (1) extend `FIXED_MODES` into a dynamic, role/permission-filtered pill set (could reuse the same capability-tree service at its root level), (2) verify a "Locked Composer" messaging pattern ("Step 1 of 2...") exists in the UI copy — not confirmed in the 80-line excerpt read. ## Study 29 — Dynamic Mode Pills & Option Selector Engine ("Human Intent Engine")

### 29.1 Vision, Fundamental Principle & 80/20 Rule (lines 12045–12066)
- **Understanding:** Reframes Mode Pills/Option Selectors as a "Human Intent Engine" translating human intent into machine-executable context for VERI, worker agents, calculation engines, and business-rule engines. Core rule: capture ≥80% of intent through structured clicks, ≤20% through free text — "context engineering, not prompt engineering."
- **Architecture/Schema implications:** Implies the resolved selection path should serialize into a structured `context` object/JSON attached to the task, with free text as a supplementary field, not the primary intent carrier.
- **Gap vs. current repo:** The `VeriComposer.tsx` `PathSegment[]` → task creation flow structurally matches this: the resolved path is what's sent, with text as an addition. Whether the ratio actually holds in practice (i.e., whether most flows genuinely resolve to a leaf without needing free text) wasn't verifiable from static reading alone.
- **Implementation recommendation:** No new build; recommend instrumenting the existing composer to log selection-path completion rate vs. free-text-reliance rate to validate the 80/20 goal empirically once in production use.

### 29.2 Progressive Discovery & Adaptive Intelligence Principles (lines 12068–12116)
- **Understanding:** Questions must be asked one at a time, each answer determining the next (not a giant form), and the entire pill/selector set must adapt to product, org, industry, role, permissions, device, and prior behavior — no static taxonomy anywhere in the system.
- **Architecture/Schema implications:** Requires the capability tree to be filterable per session by role/permission/industry — i.e., tree nodes need visibility predicates, not just a flat hierarchy.
- **Gap vs. current repo:** The tree-walk mechanism in `VeriComposer.tsx` (`nodeChildrenAt`) is structurally progressive-discovery-correct (only computes the next row's children from the path so far). Whether `capability-tree-service.ts` filters by role/permission/industry, versus returning the same tree to everyone, was not confirmed in this pass — worth a follow-up read of that service file specifically.
- **Implementation recommendation:** Read `src/lib/services/capability-tree-service.ts` in a future pass to confirm role/permission filtering exists; if it returns a static tree, that's a concrete gap against this principle.

### 29.3 Business Context Principle & Execution Graph Principle (lines 12118–12152)
- **Understanding:** Every selection builds toward a structured "Business Context" (product/customer/period/etc.) that becomes the single source of truth for downstream processing, and collectively the selections form a machine-readable "Intent Graph" consumed by VERI, the AI Orchestrator, Worker Agents, calculation/automation/business-rule/permission/audit engines — i.e., the UI is generating structured input for the backend engine mesh, not just collecting form data.
- **Architecture/Schema implications:** This effectively describes a canonical `business_context` schema (entity type + entity id + period + approval level + priority + output format) that should be the payload shape passed to the 247-engine registry and worker-agent dispatch API.
- **Gap vs. current repo:** The `PathSegment`/`CapabilityNode` resolution in `VeriComposer.tsx` (`resolveLeaf`) does pull a real leaf node (with real ids, e.g., a Project's actual id) before dispatch — this is a genuine "Intent Graph" instance, not simulated. Whether this same structured context is passed through to the 247-engine registry (`src/lib/engines/`) or the `/api/ai/team/dispatch` worker-agent API specifically was not verified in this pass — the composer sends to `POST /api/tasks`, and whether that task payload is what ultimately reaches the engines/agents needs a dedicated trace.
- **Implementation recommendation:** Trace `POST /api/tasks` → task-service.ts → downstream consumers to confirm the resolved path/context actually reaches the engine registry and dispatch API end-to-end, rather than being logged/stored without being consumed. This is a good target for a future implementation-verification pass (not part of this study-only task).

### 29.4 Prediction, Learning, Recognition-over-Recall, Minimal Cognitive Load Principles (lines 12154–12196)
- **Understanding:** VERI should predict the user's likely next selection (surfacing it first) based on history/season/org behavior; the interface should get simpler over time as it learns, never more surprising; users should recognize options rather than recall commands; every step should reduce — never add — cognitive load (less thinking/searching/typing/scrolling).
- **Gap vs. current repo:** No evidence found of predictive reordering or frequency-based promotion of tree nodes in the code read; the tree walk appears to render nodes in a fixed/service-returned order. This aligns with the "learning behaviour" gap already flagged in 28.7.
- **Implementation recommendation:** Same recommendation as 28.7 — this is one feature (usage-frequency-driven node ordering) that would satisfy multiple spec principles across Study 28 and 29 simultaneously; worth prioritizing as a single build item rather than treating each principle separately.

### 29.5 Conversational Guidance, Context Completion/Validation Principles (lines 12197–12222)
- **Understanding:** VERI should narrate the selection process conversationally ("Great, we're working on Finance," "I found three active GST registrations...") rather than presenting a bare mechanical form; the free-text composer stays locked until VERI determines sufficient context exists; before execution, VERI validates context (missing customer, invalid period, insufficient permissions, conflicting selections) to catch errors pre-execution.
- **Gap vs. current repo:** The `VeriComposer.tsx` code (per the excerpt read) is structurally form-like (rows of chip options) rather than narrated with inline conversational copy per step — no evidence of dynamic sentence generation like "I found three active GST registrations" tied to live data counts. Pre-execution validation (locked send until leaf resolved) does exist structurally, but permission/period-conflict validation specifically wasn't confirmed.
- **Implementation recommendation:** Plan-level: layer short contextual copy (data-driven, e.g. "Found {n} matching records") above/beside each selector row, sourced from the same query that populates the row's options — incremental UI work on top of the existing composer, no new backend needed if counts are already available from `capability-tree-service.ts`.

### 29.6 Personalization, Industry Adaptation, Product/Module Adaptation Principles (lines 12224–12262)
- **Understanding:** Different roles (CFO vs PM vs HR Manager) should see different pill sets; the entire pill/selector system should read as industry-specific (construction vs healthcare vs retail, etc.) without redesign; and the system must auto-evolve as products/modules are installed/removed, with zero redesign needed.
- **Gap vs. current repo:** Given the repo already runs both VERIDIAN (generic multi-tenant) and PROJEXA (construction-specific, per memory: BOQ/site-diary/manpower modules) via the same `/api/v1/projexa/*` aliasing layer, there is real infrastructure for multi-product adaptation at the API level. Whether that propagates into role/industry-specific pill sets in `VeriComposer.tsx` specifically (vs. only affecting page routing/dashboards) is unconfirmed — the composer's `FIXED_MODES` being hardcoded to 3 generic modes (discuss/chats/todo) suggests the pill layer itself is not yet industry/role-adaptive, even if the *option tree underneath* may pull from PROJEXA-specific capability nodes when in a PROJEXA workspace.
- **Implementation recommendation:** Confirm whether `capability-tree-service.ts` returns different root nodes per active product (VERIDIAN vs PROJEXA) and per role; if it already does, the main remaining gap is just extending the pill layer itself to be dynamic (same fix as 28.4).

### 29.7 Zero Dead-End, Confidence, Future-Proof & Final Principles (lines 12264–12292)
- **Understanding:** Users should never hit a screen with no clear next action; the UI should continuously reinforce user confidence (always visible: where am I, what does VERI understand, what's needed next); the entire interaction model must scale from 1 module/user/product to unlimited without redesign. Final line frames Mode Pills/Option Selectors as "the cognitive bridge between human intent and intelligent execution," judged by how little users need to think.
- **Gap vs. current repo:** Structurally scalable given the tree-driven (not hardcoded-per-module) design of `VeriComposer.tsx` — this is a genuine architectural strength already in place. "Zero dead-end" and "confidence" are UX-polish principles that would need direct usability testing to verify, not static code reading.
- **Implementation recommendation:** No code gap identified beyond what's already flagged (mode-pill dynamism, predictive ordering, conversational narration). Recommend a usability pass once those are built, rather than new architecture.

- **Gap vs. current repo:** Radius tokens already exist and are close: `globals.css` defines `--radius-sm: 6px`, `--radius-md: 10px`, `--radius-lg: 16px`, `--radius-2xl: 28px`, `--radius-full: 9999px` — same *concept* (a graduated radius scale ending in a full-pill value) but different specific numbers than spec's 8px/12px/999px. Button component (`src/components/ui/button.tsx`, shadcn-based) likely already has variant support for subtle vs. destructive styling structurally, though current primary color is saffron/orange rather than the "white with subtle border" default the spec wants. Shadow utilities (`shadow-saffron`, `shadow-saffron-fab`) are explicitly *not* "almost invisible" — they're branded, colored, elevated shadows, a direct conflict with this principle. No 8px-grid spacing enforcement mechanism (e.g., a lint rule or Tailwind spacing-scale restriction) was found — spacing appears to rely on default Tailwind scale, which is 4px-based and broadly compatible but not restricted to the specific eight values listed.
- **Implementation recommendation:** Radius: minor value adjustment if rebrand approved. Shadows: would need new, much softer shadow utilities to replace `shadow-saffron*`. Spacing: could add a Tailwind theme restriction or a lint rule enforcing the 8-value scale, but likely not worth the churn unless design-system discipline becomes a stated priority.

## Cross-Cutting Synthesis

Themes that recur across multiple parts, pulled together here rather than repeated per-subsection.

### 1. The single most repeated gap: no Enterprise Cognitive Graph

Nearly every part independently proposes its own graph — Enterprise Intent Graph (Part 2), Enterprise Conversation State Graph (Part 2), Wisdom Graph / Innovation Graph / Foresight Graph (Part 5), plus the base "Universal Cognitive Graph" and "Knowledge Objects" concepts in Part 1. None of these exist. The repo uses relational tables plus a generic `embeddings` table (pgvector-backed, per prior session notes) instead of any graph structure. This independently reconfirms a finding from the user's own prior "AI OS Certification" audit: no real knowledge graph exists in the platform today, despite being claimed in places.

This is the single highest-leverage foundational decision blocking the most other proposals in the document. **Recommendation for the eventual joint implementation plan:** build one generalized graph substrate — a single `entityRelationships` edge table (`fromType, fromId, toType, toId, relationType, weight, metadata`) in the existing Supabase Postgres — rather than the ~5 separate graph systems the document proposes piecemeal. Every part's graph proposal (intent, state, wisdom, innovation, foresight) can be a typed subset of the same table.

### 2. No Intent Engine anywhere, and it blocks almost everything downstream

Confirmed absent in every part that touches it (Parts 1, 2, 6). `chat-service.ts` sends user text straight to the LLM with no structured-intent extraction step. Nearly every "software vs. AI" routing decision, every state-machine transition, every template/component selection described across CSV 201–211 presupposes a structured intent object exists first. This is the correct *first* build priority if/when implementation begins — before CKB, before CCL, before the state machine — since those all consume intent objects as input.

### 3. Real, working infrastructure that should be extended, not rebuilt

The study found several places where the document's vision is *already substantially real*, and the risk is duplicating rather than extending:

- **Capability Registry** (`src/lib/services/capability-registry-service.ts`) — embedding-based duplicate detection across worker agents/automation rules/modules/prompt patterns, live UI and API. Closest concrete match to Principle 5 (Universal Capability Library) found anywhere in the document.
- **Worker Agent registry** (`worker_agents` table, `worker-agent-service.ts`) — tier/lifecycle/version/usage/accuracy fields, matches Principle 11 (Worker Agent Library) closely. Distinct from, and sometimes confused with, the 27-role "AI Dev Team" (`src/lib/ai-team/roster.ts`), which is a build-time engineering org chart, not a runtime business-capability agent registry — this naming collision is worth resolving explicitly before more agent-related work is scoped.
- **FDE / capability search** (`fde-service.ts`, `capability-registry-service.ts`) — embedding-based capability search with a high-confidence zero-LLM-call short-circuit, backed by a real `fde_requests` table and UI. This is the closest thing in the entire repo to the document's "Software First, AI Last" philosophy working end-to-end, and should be the template for the "Need LLM?" gate the rest of the document keeps asking for.
- **Prompt template versioning** (`prompt-os-resolver.ts`, `promptTemplates`/`promptVersions`) — production-proven version/label promotion, a solid partial match to CSV 202's Conversation Knowledge Base versioning concept, though scoped to LLM system prompts rather than user-facing response templates.
- **VeriComposer / veri-chat-context** — a real, capability-tree-driven Dynamic Option Selector wired to the live `/api/tasks` pipeline. This was an unexpected finding: Part 6 shows Studies 28/29's hardest UI piece (structured, non-free-text intent capture) is already partially shipped, narrowing that gap considerably.
- **Weighted decision matrix** (`erpRfqScoringCriteria`/`erpRfqQuotationScores`) — real configurable-weight, multi-criteria scoring, a solid template to generalize into CSV 211's Decision Intelligence Engine rather than building one from scratch.
- **Generic workflow engine** (`approvalWorkflowDefinitions`/Steps/Instances/Approvals) plus a *second*, separate `pmsWorkflowTransitions` state machine — both real, but PROJEXA's newer modules (BOQ, punch lists, change orders) don't reuse either and hand-roll their own status enums instead. A consolidation opportunity, not a build-from-zero gap.

### 4. Token-inefficiency: a concrete, fixable near-term bug

`buildConversationHistory` in `src/lib/services/chat-service.ts` resends the full message history to the LLM on every turn. This directly contradicts the document's repeated core rule (Part 1 Study 1 Level 4, Part 2 CSV 201/206) to never resend history — use context references and IDs instead. This is the single cheapest, most concrete, most immediately actionable fix found anywhere in the six parts, and doesn't depend on any other missing subsystem.

### 5. Two confirmed false "done" claims in existing governance files

- `ai-os/boss/BOARD.yaml` marks "Populate ARTIFACTS.yaml knowledge graphs" as `completed` — but `ARTIFACTS.yaml` does not exist anywhere in the repo.
- `ai-os/OS.yaml` references `ai-os/registry/ARTIFACTS.yaml` as part of the platform's own self-description — also does not exist on disk.

These aren't gaps in the *document's* vision — they're existing inaccuracies in the *platform's own governance records* that this study surfaced as a side effect of verifying claims against real files. Worth flagging to whoever owns `ai-os/boss/BOARD.yaml` independent of anything else in this study.

### 6. The "self-improving system" claim doesn't hold up under its own data

`loopImprovements` — the table structurally closest to the document's Continuous Learning Engineering (CLEE) proposal — has zero rows, ever, despite 11 loops reportedly running daily (per `AI_OS_CERTIFICATION.md` §1.5, confirmed via live query). The self-improvement *mechanism* exists; it has never actually produced an improvement. This is a sharper, more specific version of the same finding as the graph gap: infrastructure exists, but the loop that's supposed to close never closes.

### 7. Two decisions that are product/brand calls, not engineering gaps

- **Design language fork**: the document's "VERIDIAN Design Language / Calm Intelligence Design System" specifies lavender (#7C6CF2) as VERI's signature brand color. The live product's actual identity (`CLAUDE.md`, confirmed in `globals.css`/`tailwind.config.ts`) is Navy #1C2B3A / Saffron #F5820A / Teal #0E7C6E / Cream #FFFDF9. These are two different, complete, self-consistent brand systems — not a partial gap to fill in, but a fork point requiring an explicit decision (adopt lavender as an AI-specific accent within the existing palette? replace the palette? keep both separate?).
- **Conversation model**: CSV 206 (Part 2) envisions many parallel, nested, stateful workflow conversations per user. The live product deliberately gives each user exactly one persistent AI thread (`ensureAiThread` singleton pattern). Moving toward the document's model is a real product-UX decision (surface multiple visible threads? keep one thread with hidden parallel state underneath?), not something to default into while building the state machine.

### 8. Process fix that closes several gaps at once, cheaply

Principles 14 and 17 (AI Coding Directive, Platform Evolution Principle) require checking the Capability Registry before writing new code, and generalizing successful customer-specific work back into the platform. The *tooling* for the first already exists (`findSimilarCapabilities()`/`auditDuplicateCapabilities()`) but isn't mandated anywhere — no CI step or PR checklist item requires it. Making this mandatory in `AGENTS.md`/PR template is close to free and would materially reduce future duplicate-capability risk without any new subsystem.

## Addendum: Proposed VERIDIAN AI OS Multi-Repo "Brain" Architecture (added 2026-07-09, per Boss request)

The repo owner asked directly: how should GitHub repositories be architected so that VERIDIAN AI OS — a self-learning, self-evolving, self-developing cognitive AI OS, "THE BRAIN" — stays structurally separate from modules/functionalities/projects/products, while controlling every FChecklist repository, with modules free to live in their own repos, everything connected by APIs, full reuse with no duplication, a directory that supports "what do we already have" analysis before any new development, and a full AI agent team running it? This addendum answers that directly, building on the gap analysis above (this is a proposal, not something implemented in this PR — no code changes are made here either).

### 1. Why the current repo topology doesn't satisfy this yet

Today, the "brain" (Capability Registry, Worker Agent registry, the 25-file deterministic engine library, FDE capability search, prompt/conversation infrastructure) lives inside `compliance-tracker`'s `src/lib/` — the same codebase as VERIDIAN's own compliance product logic. PROJEXA reuses it only because it's aliased into the same Next.js process (`/api/v1/projexa/*`), not because it's calling a genuinely separate, independently-versioned service. `veda-advisors` is a separate repo but has no wiring to any of this intelligence at all. There is no single place a new repo, or a new feature inside an existing repo, can query to ask "does this capability already exist anywhere in FChecklist?" — the registry that exists is trapped inside one product's codebase. This is the structural reason the Constitution's Principle 14 (AI Coding Directive — "check before you build") and Principle 16 (Universal Asset Registry) can't yet be enforced platform-wide: there is no platform-wide anything, only one big repo and a couple of unconnected smaller ones.

### 2. Proposed repo topology

**`FChecklist/veridian-brain`** — THE BRAIN. A new, standalone repo containing only cognition and governance, no business/product logic:
- Capability Registry + Asset Registry (generalizing the existing `worker_agents` schema shape — tier/version/lifecycleStatus/usageCount/qualityScore — into the shared `platform_assets` table proposed in Phase 2 above, covering modules, worker agents, integrations, APIs, templates, forms, dashboards, reports, prompt patterns, and every other category Principle 16 lists)
- Worker Agent registry + runtime dispatch (today's `worker_agents` table, `worker-agent-service.ts`, and the `/api/ai/team/dispatch` pattern, generalized to serve every repo, not just `compliance-tracker`)
- The deterministic engine library (today's 25-file `src/lib/engines/`, the `computation_engines` tracking table, and FDE's embedding-based capability search — `fde-service.ts` — which is already the closest real match to the document's "Software First, AI Last" philosophy found anywhere in the study)
- Intent Engine, Conversation OS/Knowledge Base/State Machine (net-new — see Phase 2/3 above)
- The Enterprise Cognitive Graph (`entityRelationships` edge table, per the cross-cutting synthesis above) — the one graph substrate serving Intent/State/Wisdom/Innovation/Foresight graph use cases
- Model Router / LLM Gateway (today's `llm-client.ts`, OpenRouter integration, `llm-response-cache.ts`)
- SENTINEL governance (today's `ai-os/sentinel/`), but scoped platform-wide instead of per-repo
- A cross-repo `BOARD.yaml`/task board that every module repo's local board syncs into, so the Brain has a single global view of what's in flight everywhere — this is the literal "VERIDIAN AI OS is the boss" mechanism: the Brain repo is where cross-repo work gets analyzed, approved/rejected/modified, and tracked, mirroring the single-gateway pipeline the source document's own opening instructions call for

**Product/module repos** (`compliance-tracker`, `projexa`, `veda-advisors`, and future ones — e.g. a future dedicated HRMS or CRM module) — each stays a normal application repo: its own UI, its own product-specific database tables, its own deploy pipeline. The only thing that changes is that anywhere these repos currently import `src/lib/engines/*`, `capability-registry-service.ts`, `worker-agent-service.ts`, or `fde-service.ts` directly, they instead call the Brain's API through a published SDK.

**`@veridian/brain-sdk`** — a small, versioned npm package (published from the Brain repo, consumed by every module repo) wrapping the Brain's API: typed client functions (`findSimilarCapabilities()`, `dispatchWorkerAgent()`, `classifyIntent()`, `runEngine()`, `queryGraph()`), with local response caching and retry/fallback logic built in, so module repos get a typed, ergonomic interface instead of hand-rolled `fetch()` calls. This is also where the "search before you build" directive becomes mechanically enforceable — a lint rule or CI check in module repos can require `findSimilarCapabilities()` to have been called (via the SDK, so it's traceable) before a new service file is added.

### 3. Integration contract — how repos talk to the Brain

- **Primary interface**: versioned REST API (`/api/v1/brain/*`) exposed by the Brain repo's own Next.js/API deployment, authenticated via service-to-service tokens (each module repo gets its own credential, scoped and revocable — extending the existing `worker_agents.tier` model of global/customer/client/user scoping to repo-level scoping).
- **Secondary interface**: an MCP server wrapping the same API, so AI coding agents (Claude Code, z.ai, Codex, etc.) working in *any* FChecklist repo can query the Brain's registry directly from their tool-use loop — "does this capability exist?" becomes a tool call, not a manual grep. This directly operationalizes Principle 14 for AI agents specifically, which is exactly who's doing nearly all the coding here per `AGENTS.md`.
- **Event Bus**: the Constitution (Principle 3, 15) calls for an Event Bus and currently has none anywhere in the platform (confirmed absent in Part 1 of this study). This is the natural place to add it — the Brain repo hosts a lightweight pub/sub layer (e.g., Postgres `LISTEN/NOTIFY` via Supabase realtime, or a queue) so module repos can publish events ("Invoice Approved," "Project Created") without knowing who else consumes them, and the Brain can react (e.g., trigger a worker agent, log to the knowledge graph) without the publishing module repo needing to know the Brain exists beyond "publish this event."
- **No direct module-to-module calls, ever**: this is the literal enforcement of Principle 15 (Universal Dependency Rules). If `projexa` needs something from `compliance-tracker`'s domain, it goes through the Brain (a shared capability, worker agent, or event) — never a direct API call between the two product repos. This is easy to enforce structurally once they're genuinely separate repos with separate deploy targets (today it's only a convention inside one Next.js app, which is why Part 1 of this study found it isn't actually enforced).

### 4. Registry as the pre-development analysis gate

The specific thing requested — "a proper directory where before any new development, the analysis can happen what we have and what we need" — is the Brain's Capability/Asset Registry, made mandatory rather than optional:

1. Before scoping any new module, feature, or repo, the first step is a Brain API/SDK query: `findSimilarCapabilities(description)`. This already works today as `capability-registry-service.ts`'s embedding-based search (0.92 similarity threshold) — the only change needed is making it the *required first step*, platform-wide, not a tool that happens to exist.
2. If a match exists: extend/configure it (per the Constitution's own "Build once, configure many" mantra) — no new code.
3. If no match exists: build it *inside the Brain* if it's genuinely cross-product-reusable (an engine, a worker agent, a capability), or *inside the specific module repo* if it's genuinely product-specific business logic (a compliance checklist type, a construction BOQ field) — and either way, register it in the Asset Registry on completion, closing the loop Principle 17 (Platform Evolution Principle) describes.
4. This sequence should be a literal PR template checklist item in every module repo (`AGENTS.md` already exists in `compliance-tracker` and sets a precedent for this kind of enforced-process document) and, ideally, a CI check that fails a PR if no Brain registry query is evidenced for new service-layer files.

### 5. AI agent team to run the Brain

This is distinct from the existing 27-role "AI Dev Team" (`src/lib/ai-team/roster.ts`), which — as flagged earlier in this study — is a build-time engineering org chart (Engineering/QA/Finance-style roles), not a runtime team that operates the cognitive platform itself. Recommend a separate, smaller roster scoped specifically to running the Brain day-to-day, extending the existing `ai-os/boss/ROSTER.yaml` pattern (today: only 3 infra agents — VEDABOSS, ZAI, SENTINEL):

- **SENTINEL** (exists today) — supreme governance guardian, unchanged in role, but its authority becomes platform-wide (spans every repo) instead of per-repo.
- **BRAIN-ARCHITECT** — owns the Brain repo's own internal architecture and evolution; the one role allowed to approve changes to the Brain's own schema/API contract, since a breaking change there ripples to every module repo.
- **CAPABILITY-CURATOR** — maintains the Asset Registry; runs `auditDuplicateCapabilities()` on a schedule; is the role that actually enforces step 1 of the pre-development gate above rather than leaving it to convention.
- **KNOWLEDGE-GRAPH-KEEPER** — owns the `entityRelationships` graph once built; ingests new relationships as modules report events/usage; the role most directly responsible for closing the "no knowledge graph exists" gap found repeatedly throughout this study.
- **WORKER-AGENT-MANAGER** — owns the worker agent lifecycle (propose → test → approve → publish → retire) across *all* module repos, not just one.
- **INTEGRATION-GATEKEEPER** — owns the Brain's API/SDK versioning and backward-compatibility; the role a module repo's PR pipeline effectively answers to when it changes how it talks to the Brain.
- **MODULE-LIAISON** (one instance per product repo, e.g. `PROJEXA-LIAISON`, `COMPLIANCE-LIAISON`) — lives conceptually in the module repo, is the one that calls up to the Brain, proposes new capabilities for promotion into the Brain when something built product-specific turns out to be reusable (operationalizing Principle 17).
- **PERFORMANCE-ENGINEER** — owns latency/token budget across the whole system; directly answers "make the system faster" by tuning the Model Router, SDK caching, and watching for exactly the kind of token-inefficiency bug found in this study (`buildConversationHistory`'s full-history resend).
- **EVOLUTION-ANALYST** — mines usage telemetry (worker agent usage logs, engine call volume, cache hit rates) into actual entries in the self-improvement loop table — directly targeting the gap found in this study where `loopImprovements` has zero rows ever despite loops running daily.

### 6. Making it faster, not just cleaner

Splitting into real repos risks making things *slower* (network hops replace in-process function calls) unless this is designed for deliberately:

- The `@veridian/brain-sdk` caches aggressively client-side (capability lookups, engine metadata, worker agent definitions change far less often than they're read) — most calls should resolve from a local cache, not a network round-trip, most of the time.
- High-frequency, latency-sensitive operations (the deterministic engines themselves — GST calculation, EMI, payroll math) should be *pure functions* published as a shared npm package (`@veridian/engines`) that module repos run **in-process**, not as network calls to the Brain — only the *registry/metadata about* which engines exist needs to be a network call, not every invocation of a fast deterministic calculation. This distinction (registry lookups over the network, hot-path execution in-process via a shared package) is the single most important design decision for making this architecture faster rather than slower than today's monolith.
- The Model Router / LLM Gateway is a genuinely good candidate for staying centralized in the Brain (it's already inherently a network call to an external LLM provider, so routing it through the Brain first adds negligible relative latency, while unlocking cross-repo prompt caching, cost tracking, and the "Need LLM?" gate working platform-wide instead of per-repo).

### 7. Migration path — strangler-fig, not a rewrite

Given `compliance-tracker` is a live, working system, this should not be a big-bang extraction:

1. **Phase A (inside `compliance-tracker`, no new repo yet)**: wrap `capability-registry-service.ts`, `worker-agent-service.ts`, `src/lib/engines/*`, and `fde-service.ts` behind a clean internal `/api/v1/brain/*` route namespace (even though it's still the same deploy). This proves the API contract without any cross-repo risk.
2. **Phase B**: point `projexa` and `veda-advisors` at that same `/api/v1/brain/*` namespace over the network (instead of `projexa`'s current alias-based reuse, which only works because it's not actually a separate repo yet) — this validates the contract works cross-repo before any code physically moves.
3. **Phase C**: stand up `FChecklist/veridian-brain` as its own repo/deploy, move the wrapped services there, publish `@veridian/brain-sdk` and `@veridian/engines`, and repoint all module repos at the new service. `compliance-tracker` itself becomes a module repo at this point, on equal footing with `projexa` and `veda-advisors`, rather than the "flagship that happens to also host the brain."
4. **Phase D**: build the net-new Brain components that don't have a today-equivalent to migrate (Intent Engine, Conversation OS/State Machine, Enterprise Cognitive Graph, Event Bus) directly inside the new Brain repo, per the phased plan in the Cross-Cutting Synthesis above.

This mirrors the Constitution's own "Build once, configure many, reuse everywhere, govern centrally, improve continuously" mantra (Principle 19) applied to the migration itself, not just to the end state.
